# WS4.0 — Analysis: can `Inert` be stored inline, and what breaks?

**Date:** 2026-07-09 · **Author:** WS4 session (maintainer-authored deliverable per roadmap 4.0) · **Gates:** G1 (decided in direction) · **Feeds:** WS4.1 (go/no-go), WS5.1 (Layout design input)

This is the written deliverable the roadmap's 4.0 item requires before 4.1 may land. It answers: **(1)** can the runtime-node-backed `Inert` be replaced by an inline value, **(2)** what is the full Copy-loss blast radius (G1), **(3)** is the `Layout` scope-cut to WS5.1 justified, and **(4)** the go/no-go for 4.1.

Evidence gathered by two adversarial code sweeps over the whole workspace (`rsact-reactive`, `rsact-ui`, `rsact-render`, `rsact-macros`, `metrics-probe`, `src/`); all claims are `file:line`-cited.

---

## 0. What "inline `Inert`" means (recap of today's shape)

- `Inert<T>` (`rsact-reactive/src/inert.rs:13`) = `{ id: ValueId, PhantomData }` — a `Copy` **handle** to a `ValueKind::Stored` runtime node minted by `rt.create_inert` (`runtime.rs:587`, identical to `create_stored`). Read-only (`ReadSignal` only; **no `WriteSignal`** — `inert.rs:55-66`).
- `MaybeReactive<T>` (`maybe/maybe_reactive.rs:47`) = `Inert(Inert<T>) | Memo(Memo<T>)`, **unconditionally `Copy`** (`:64`) because both arms are id handles.
- `Memo<T>` (`memo.rs:96`) = `Inert(Inert<T>) | Memo{id} | Signal(Signal)`, unconditionally `Copy` (`:129`).
- The precedent we are mirroring already exists: `MaybeSignal<T>` (`maybe/maybe_signal.rs:51`) stores `Inert(Option<T>)` **inline** and is `Copy` **iff `T: Copy`** (`:69`). WS4.1 applies that same shape to `Inert`/`MaybeReactive`.

The target (roadmap sketch): `struct Inert<T>(T)` (zero runtime presence); `MaybeReactive<T>` stores the inert value inline; `Copy` iff `T: Copy`, `Clone` otherwise (G1).

---

## 1. Is inlining `Inert` semantically safe? — **YES.**

An `Inert<T>` is a **read-once, immutable, unaliased-for-mutation** value. Replacing the node handle with `Inert<T>(T)` is observationally identical:

- **Never mutated through its handle.** No `WriteSignal` impl exists for `Inert` anywhere (verified by grep). `track()` is a no-op (`inert.rs:63-65`).
- **Never upgraded in place.** `Runtime::make_reactive` (the stored→signal in-place upgrade, `runtime.rs:631`) has **zero call sites** in the entire workspace — only its definition and one doc-comment mention (`signal.rs:205`). The reactive-on-write upgrade path that CLAUDE.md describes as "the basis of MaybeSignal/MaybeReactive" is, in practice, **dead code for `Inert`**: `MaybeSignal::now_reactive` (`maybe_signal.rs:184`) mints a *fresh* signal and swaps the enum; `MaybeReactive` is read-only. Nothing calls `make_reactive` on an `Inert`'s id.
- **Aliasing is observationally invisible.** `Inert: Copy` means two handles share one `ValueId`, but since the value is immutable and read-only, sharing the node vs. duplicating the value is indistinguishable to every observer.
- **The read-trait macro needs only `with_untracked`.** `impl_read_signal_traits!(Inert<T>)` (`inert.rs:25`) generates `PartialEq`/`PartialOrd`/`Neg`/`Not`/arithmetic (`read.rs:262-323`), all expressed through `self.with(|v| …)`. None reference a `ValueId`. Swapping `with_untracked` to `f(&self.0)` keeps every one compiling unchanged; bounds (`T: PartialEq + Clone + <op>`) are unaffected.
- `get()` needs `T: Copy` (`read.rs:56`), `get_cloned()` needs `T: Clone` (`read.rs:72`). The acceptance test `static_wrapper` (`maybe/mod.rs:39`) reads `123.inert()` (i32, Copy) via `get`/`get_cloned`/`get_untracked`/`map` — **all pass trivially once no node is created**.

**Follow-up (minor):** after inlining, `Inert::id()` must return `None` (mirroring `MaybeSignal::Inert`'s `id → None`, `maybe_signal.rs:76`), not `Some(id)` (`inert.rs:42`). Delegated `Memo::id`/`MaybeReactive::id` for an inert therefore return `None`; no mutation or lifecycle path depends on that id (confirmed — the only stored-node mutation in the tree is `Layout`, which uses a raw `ValueId`, not `Inert<T>` — see §3).

## 2. The Copy-loss blast radius (G1) — small, and the one sharp edge is avoidable

Dropping blanket `Copy` from these types has three distinct blast zones. Only **two** non-`Copy` inner types are ever put in a `Memo`/`MaybeReactive` in the whole workspace: `String`/`Vec<Layout>` (in the layout structs) and `LayoutModel`/`Vec<MemoTree<T>>` (in `Memo`). Everything else (`f32`, `u32`, `bool`, `Size`, `FontSize`, …) is `Copy` and unaffected.

### 2a. `Inert<T>` copy-loss — trivial (2 sites)

Only two sites copy an `Inert<T>` out of a *borrow* where `T` has no `Copy` bound:
- `page/mod.rs:666` — `let stylist = self.stylist;` inside `use_renderer(&mut self)`, `stylist: Inert<W::Stylist>`.
- `ui.rs:293` — `self.stylist` passed by value into `Page::new` from a `&self`.

`InternalStylist` (`style/stylist.rs:25`) has no `Copy`/`Clone` supertrait, so the *generic* code needs a fix even though concrete stylists (`()`, `BinaryTheme`) are `Copy`. **Fix:** add `W::Stylist: Copy` (or `Clone` + `.clone()`) — one bound. (`ui.rs:157` is a *move* out of owned `self`, not a reliance.)

### 2b. `MaybeReactive<T>` copy-loss — the "layout Copy tower" (**forced, unavoidable, contained**)

Four layout payload structs `#[derive(Copy)]` while embedding a `MaybeReactive<NonCopy>` **directly** (not behind a node):
- `ContentLayout` (`layout/mod.rs:63`) — `content: MaybeReactive<String>` (`:67`).
- `FlexLayout` (`layout/mod.rs:240`) — `children: MaybeReactive<Vec<Layout>>` (`:249`).
- `LayoutKind` (`layout/mod.rs:532`) — contains the above.
- `LayoutData` (`layout/mod.rs:542`) — contains `LayoutKind`.

These four **must drop `Copy` and keep `Clone`** (every field is `Clone`: `String`, `Vec<Layout>`, `Layout` are all `Clone`). Plus one by-value read to fix: `match self.kind {` (`layout/mod.rs:603`) → `match &self.kind`. Siblings already use `match &self.kind` / `&mut self.kind` (`:565,572,586,614,633,652,662,677`).

**This is inherent to the WS4 goal, not a choice.** `ContentLayout` holds the (possibly static) label text; keeping it node-free for a static label *requires* an inline `MaybeReactive<String>`, which cannot be `Copy`. The only way to keep `ContentLayout: Copy` is to keep the text node-backed — which reintroduces exactly the node WS4 exists to kill. So the tower collapse is the price of node-free static text.

**Crucially, `Layout` itself is NOT touched.** `Layout` (`layout/node.rs:18`) = `Static(ValueId) | Reactive(Signal<LayoutData>)` — both variants are id handles, so `Layout` stays `Copy` regardless of `LayoutData`'s derives. The breakage stops at `LayoutData`/`LayoutKind` and does **not** cascade through `Layout`. `ContainerLayout`/`ScrollableLayout` hold `content: Layout` (still Copy) and are unaffected.

Widgets that hold `MaybeReactive<NonCopy>` fields (`Label` `content: MaybeReactive<String>` `label.rs:19`; `Slider` `range`/`step`; `Select` `options`; `Bar` `value`; `Icon`) are all `#[derive(View)]` structs — `#[derive(View)]` does **not** emit `Copy` (verified in `rsact-macros`), and they are *moved* into the arena, never copied. **Unaffected.**

### 2c. `Memo<T>: Copy` becoming conditional — the sharp edge, **avoided by removing `Memo::Inert`**

If we naively inline `Memo::Inert(Inert<T>)`, then `Memo<T>: Copy` becomes `T: Copy`-conditional, which detonates two severe sites:
- `memo.rs:267` — `impl Copy for MemoTree<T>` becomes uncompilable for **every** `T` (its `children: Memo<Vec<MemoTree<T>>>` can never be `Copy`).
- `page/mod.rs:665` — `let layout = self.layout;` on `Memo<LayoutModel>`. `LayoutModel` (`layout/model.rs:83`) is neither `Copy` **nor `Clone`**, so `Memo<LayoutModel>` would be neither — **unrescuable by `.clone()`** without either making `LayoutModel: Clone` or restructuring.
- Plus `memo.rs:205` (`let this = *self`) and `memo.rs:285` (`flat_collect` copies `self.data`).

**Decision: keep `Memo<T>` unconditionally `Copy` by removing the `Memo::Inert` variant entirely.** Rationale, backed by the sweep:
- `Memo::Inert` is produced by exactly one place: `IntoMemo for Inert` → `Memo::Inert(self)` (`inert.rs:83`), reached transitively via `IntoMemo for MaybeReactive` (`maybe_reactive.rs:290`).
- `MemoTree` is **not used anywhere in `rsact-ui`** (only the reactive crate's own tests + commented-out code at `layout/mod.rs:747` and `rsact-encoder`). So the whole `MemoTree`/`Memo::Inert` interaction is confined to `rsact-reactive`.
- The only live `.memo()` calls in `rsact-ui` are `icon.rs:78` (`Signal<FontSize>` → `Memo::Signal`) and `show.rs:25` (`bool`) — **neither constructs a `Memo::Inert`, and both `T` are `Copy`.** No hot UI path converts a constant to a memo.
- After removal, `Inert::memo()` / `MaybeReactive::memo()` (inert arm) create a real constant memo node (`create_memo(move || v.clone())`, adding a `T: Clone` bound already satisfied by the `MaybeReactive::memo` caller at `maybe_reactive.rs:286`). This allocates a node **only** when code explicitly demands a `Memo` from a constant — a rare, honest cost that is **not** the builder-literal path (builder literals stay `MaybeReactive::Inert`, inline, node-free).

With `Memo::Inert` gone, `Memo<T>` stays `Copy`/`Clone` for all `T`, so **`MemoTree`, `page/mod.rs:665`, `memo.rs:205/285` remain untouched.** This is the decisive simplification: it removes the entire §2c blast zone at the cost of one cold allocation path.

### Blast-radius summary

| Zone | Sites | Severity | Resolution |
|---|---|---|---|
| `Inert<W::Stylist>` copy-out | 2 (`page/mod.rs:666`, `ui.rs:293`) | trivial | add `W::Stylist: Copy` bound |
| Layout Copy tower (`MaybeReactive`) | 4 derives + 1 match (`layout/mod.rs:63,240,532,542,603`) | mechanical, **forced** | `Copy`→`Clone` + `match &self.kind` |
| `Memo<T>: Copy` conditional | (avoided) | would be severe | **remove `Memo::Inert`** |

Everything else the compiler finds will be `Copy`-inner and compile unchanged (the `MaybeSignal::Inert` precedent proves the pattern already lives in-tree with no wider fallout).

## 3. Is the `Layout` scope-cut to WS5.1 justified? — **YES.**

`Layout` genuinely needs **shared identity** and cannot be inlined like a plain constant:

- **It is a shared handle by design.** A child widget keeps its own `Layout` field *and* hands a Copy of the same handle to its parent's collection: `FlexLayout.children` (`flex.rs:45-47`), `ContainerLayout.content` (`container.rs:24-26`), `button.rs:29`, `scrollable.rs:108`, `select.rs:188`, and the arena's `El::Stored { layout }` (`arena.rs:249`). The layout pass reads through the **parent-held** copies (`model.rs:263,327,367,389`) while builder setters mutate through the **widget-held** copy (`widget/mod.rs:119-238`). They agree only because `Copy` preserves the `ValueId`. An inlined by-value `Layout` would give each holder an independent copy → stale layout. Inlining is *semantically wrong* here, the opposite of the read-once `Inert` props.
- **`Copy` is also the cause of the `LayoutMut` bug class.** `Layout::now_reactive` (`node.rs:48-69`) disposes the old `Static` id and reassigns `*self` to a `Reactive` signal; done through a throwaway copy (`self.layout()` returns by value, `widget/mod.rs:61`), the disposal lands but the reassignment is lost → panic on next read. The codebase already had to contain this with the `layout_mut` supertrait (`widget/mod.rs:75`) and a dedicated regression test (`page/mod.rs:1808`, `EVOLUTION.md:45`).

So WS4 does the right thing by **not** touching the `Layout` handle mechanism. (Note the §2b derive changes touch the layout *payload* structs' `Copy` derives — see the scope clarification below — but not the `Layout` handle type, its `now_reactive`/`layout_mut` machinery, or the WS5.1 redesign.)

### WS5.1 design input (deliverable output for the next workstream)

The off-graph shared handle must satisfy:
1. **Shared identity by construction, off the graph:** `Layout(Rc<RefCell<LayoutData>>)`; cloning shares the same `RefCell` so widget field, parent children-vec, and arena `El::Stored` observe one datum without `ValueId` equality.
2. **Lose-proof mutation:** interior mutability + no `Copy`/mutating-`Clone` retires the `now_reactive`/`layout_mut` trap; a reactive binding is one effect writing `borrow_mut()` + marking a page dirty set.
3. **`Widget::layout` → `&Layout`** (`widget/mod.rs:61`): blast radius is 19 `layout()` impls + inherent `El::layout`, ~12 by-Copy call sites, the `layout_mut` trait + 10 impls, 3 setter traits, ~16 widgets with owned `layout` fields, and the `*layout`/`*content` deref-copies in `model.rs`/`el/mod.rs`/`dynamic.rs`.
4. **Break the layout structs' `Copy` dependence** — already done mechanically by WS4.1's §2b (they become `Clone`); WS5.1 inherits that.
5. **Explicit node identity (`ElId` recorded at build)** decoupled from `ValueId`, making the arena↔layout structural-parallel invariant explicit rather than positional.

## 4. The 4.1 design (what changes, what stays)

**INLINE** `Inert<T>` → `Inert<T>(T)`: `From<T>` stores inline (no `create_inert`); `with_untracked` → `f(&self.0)`; `track` stays no-op; `id → None`; `is_alive → true`; `dispose → drop`; `SignalMap::map` → `Inert(map(&self.0))` (no node — this is the label double-node fix, §5). `Copy` iff `T: Copy`, `Clone` iff `T: Clone`.

**INLINE** `MaybeReactive<T>`: `Inert` arm carries the inline value (keep the `Inert<T>` wrapper to minimize match-arm churn — `MaybeReactive::Inert(Inert<T>)` still works, just node-free). `Copy` iff `T: Copy`.

**REMOVE** `Memo::Inert`: `Memo<T>` = `Memo{id} | Signal(Signal)`, stays `Copy`/`Clone` for all `T`. `IntoMemo for Inert`/`MaybeReactive` allocate a constant memo node (`T: Clone`).

**LAYOUT PAYLOAD (forced §2b):** `ContentLayout`/`FlexLayout`/`LayoutKind`/`LayoutData` drop `Copy`, keep `Clone`; `layout/mod.rs:603` → `match &self.kind`. **`Layout` handle type + WS5.1 redesign untouched.**

**STYLIST (§2a):** add `W::Stylist: Copy` bound (or `Clone` + `.clone()` at `page/mod.rs:666`, `ui.rs:293`).

The compiler enumerates the rest (G1: "the compiler finds every site"); all remaining sites are `Copy`-inner.

## 5. Predicted node-count impact (re-baseline target)

Current baseline (`metrics-probe -- record` @ `9a09b7e`):

| Scenario | total | signals | memos | observers | **stored (Inert)** |
|---|---|---|---|---|---|
| reactive_only_16 | 33 | 16 | 0 | 17 | 0 |
| ui_labels_5 | 32 | 11 | 6 | 6 | **9** |
| ui_labels_10 | 52 | 16 | 11 | 11 | **14** |

`reactive_only` uses no `Inert` → unchanged (33). For `ui_labels`, the `stored` count is a mix of (a) `Layout::Static` nodes (raw `ValueId` — **WS4 does not remove these**; WS5.1 does) and (b) genuine builder-literal/prop `Inert` nodes that 4.1 inlines. The realized drop will be measured in session 1 and re-baselined in `metrics-probe/src/scenarios.rs::node_count_regression` (old→new recorded in the roadmap). The roadmap's "static page ≈ 0 nodes" is a *WS4+WS5* target; WS4 alone removes the prop/literal `Inert` nodes and (via §2c) does not add any.

## 6. GO / NO-GO

**GO for 4.1** — with the design in §4. Inlining is semantically safe (§1); the Copy-loss is small and, with `Memo::Inert` removed (§2c), confined to the forced layout-payload tower (§2b) plus two stylist bounds (§2a); the `Layout` handle and WS5.1 redesign are untouched (§3).

### Scope clarification to flag (roadmap protocol: report, don't silently fix)

Roadmap 4.1 says *"`Layout` untouched (4.0 scope cut)."* The analysis shows this holds for the **`Layout` handle type**, its `now_reactive`/`layout_mut` machinery, and the WS5.1 `Rc<RefCell>` redesign — all untouched. But inlining `MaybeReactive` **forces** the layout *payload* structs (`ContentLayout`/`FlexLayout`/`LayoutKind`/`LayoutData`) to change their `Copy` derive to `Clone` (§2b), because they embed `MaybeReactive<NonCopy>` directly and node-free static text is impossible otherwise. This is mechanical (derive swap + one `match &`), does not pre-empt WS5.1 (WS5.1 inherits it), and has no alternative short of abandoning the static-text win. **Session 1 will make this change and report it; if the maintainer intends `layout/*.rs` to be strictly untouched until WS5.1, 4.1 cannot deliver node-free static widgets and must be narrowed.**
