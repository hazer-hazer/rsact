# WS9b — deep engine surgery: design & decision gate

**Status: BLOCKED on maintainer decision(s). No surgery performed yet.** This
branch (`ws9b-engine-surgery`) carries only this design doc. It sits on top of
the WS9a work (PR #5) because 9b reworks the same `runtime.rs`/`storage.rs`
surfaces 9a touched; once #5 merges this rebases onto master.

WS9b is the two highest-risk core changes in the whole roadmap:

- **9b.1 — drop the per-value `Rc`** (leptos-style take-value-out-during-update)
  and fold the edge `SecondaryMap`s into the `Value` slot. Target: per-value RAM
  150–250 B → **< 100 B**.
- **9b.2 — pull-phase iterativization**: iterative `Check`-walk + incremental
  height maintenance + a height-ordered recompute queue. The current recursive
  `maybe_update` is a **correctness hazard** (~2 KB/level stack) on 4–8 KB
  embedded stacks. Acceptance gate: a `DEPTH=10_000` memo-chain test in a
  stack-limited thread + full criterion A/B.

## 1. The WS4 gate — verdict: SOFT (verified), but see the recommendation

The roadmap gates 9b "after WS2 + WS4". **WS2 is landed; WS4 is entirely
unstarted.** A design workflow (3 investigators + 2 adversarial verifiers) was
run to determine whether that dependency is *hard* or *soft*. Verdict: **SOFT —
9b can be implemented and measured correctly without WS4**, adversarial
refutation failed. Evidence:

- **Disjoint surfaces.** WS4 rewrites `inert.rs` / `maybe/maybe_reactive.rs` /
  `memo.rs::Memo::Inert` and rsact-ui node counts (it changes the *population*
  of runtime nodes + drops blanket `Copy`). 9b rewrites the *representation and
  pull algorithm* in `storage.rs`/`runtime.rs` (`Value`, the edge maps,
  `maybe_update`). Of WS4's items only **4.6** (capacity metrics) reads the
  structures 9b reworks, and the roadmap already flags the coordination
  symmetrically (L432: "Coordinate with 9b.1 …").
- **No correctness coupling.** 9b.1's take-out only touches callback-bearing
  kinds; inert `Stored` values run no callback, so their degrade surface is
  identical with or without WS4's inlining.
- **WS4-invariant acceptance metrics.** 9b.1's target is per-node bytes
  (node-count-invariant); 9b.2's gate is a pure `Memo` chain (zero inert nodes
  for WS4 to remove) + criterion reactive A/B. WS4 cannot change either.

**So the block is not technical.** The reasons to still treat 9b as
decision-gated are process/risk, not compilation:

- 9b is sequenced *two layers* past WS4 in the execution order; starting it now
  is a large jump ahead of the plan-of-record.
- Two highest-risk core surgeries unshipped simultaneously (9a in PR #5, 9b
  here) increases integration risk.
- **Aggregate-metric misattribution (soft):** measuring 9b.1's live-heap win on
  a *UI scenario* before WS4 over-credits it for shrinking inert `Stored` nodes
  WS4 will delete outright. Mitigation: report 9b.1 on **per-node bytes** +
  synthetic-graph terms (as the roadmap specifies), and rebaseline any
  UI-scenario aggregate after WS4.

## 2. 9b.1 — drop the per-value `Rc`  (has its own design fork)

**Today:** `Value { value: Rc<RefCell<dyn Any>>, kind, state, height }`
(`storage.rs`). Every read clones the `Rc` (`value_rc`, `storage.rs:476`) so the
storage borrow is released *before* the user closure runs; `update()` clones
`(f.clone(), value.clone())` (`runtime.rs`).

**Proposed:** `Value { value: Option<Box<dyn Any>>, … }`; `update()` **takes**
the box out, runs the callback (storage borrow released; a re-entrant read sees
`None` → logged degrade), then **puts it back**. −1 heap alloc + −`Rc` header
per node; fold `sources`/`subscribers` (and consider `owned`) into the slot.

### DECISION 2 (needs maintainer) — the read-path fork
The adversarial verifier surfaced a real hazard the roadmap sketch glosses:

> Today's **read** path (`try_with_untracked` → `value_rc`) hands out an owned
> `Rc` precisely so the storage borrow is dropped before the user closure `f`
> runs. `Option<Box<dyn Any>>` has **no `Rc` to hand out**, so the read path must
> either **(i)** hold the storage borrow across `f` (aliasing / re-entrancy
> hazard — a nested read of the same cell double-borrows) or **(ii)** take the
> box out on **reads too** — which puts inert `Stored`/`Signal` reads into the
> "sees `None`" degrade surface, not just callback recomputes.

This is the crux of 9b.1 and changes the engine's borrow semantics. Options:

- **(i) borrow-across-closure**: keep it simple, but re-entrant reads of the same
  value now conflict — needs an audit that no read closure re-reads its own cell
  (and a logged degrade if it does).
- **(ii) take-out-on-read**: uniform with the update path; every read is a
  take→run→put, so *any* re-entrant self-read (not just recompute cycles) hits
  the `None` degrade. Wider degrade surface, but one consistent mechanism.
- **(iii) hybrid**: `Copy`/small `Stored`/`Signal` values read by-copy out of the
  cell (no take, no borrow held); only callback kinds take-out. Keeps inert reads
  cheap and out of the degrade surface. **Likely the right answer**, but needs a
  size/`Copy` strategy per kind.

_My recommendation: **(iii)**, but this is a genuine engine-semantics decision —
I will not guess it. It should be settled (ideally written into this doc) before
any 9b.1 code._

Note: the 4.6 ↔ 9b.1 metric coordination spans **four** `SecondaryMap`s
(`sources`/`subscribers`/`owned`/`mark_seen`), not three; `mark_seen` is a
transient dedup structure unlikely to fold into `Value`.

## 3. 9b.2 — pull-phase iterativization

**Today:** `maybe_update` self-recurses over a node's sources
(`runtime.rs`), so a dependency chain of depth N costs ~N stack frames
(~2 KB/level) — a stack-overflow hazard on embedded's 4–8 KB stacks.

**Proposed (staged, per roadmap):**
1. Cheap tier: iterative `Check`-walk for clean nodes (explicit heap worklist,
   like `mark_dirty` already uses) + frame diet.
2. Incremental height maintenance + a height-ordered recompute queue.

**Acceptance gate:** a `DEPTH=10_000` linear `Memo` chain evaluated in a
**stack-limited thread** (e.g. `std::thread::Builder::stack_size(small)`), which
must overflow on today's recursive pull and pass after iterativization; plus the
**full criterion A/B** (`benches/reactivity.rs` `graph/memo_chain_depth` etc.)
before/after to prove no throughput regression.

This item has no major design fork (it's a mechanical recursion→explicit-stack
transformation), but it is high-risk (pull correctness / glitch-freedom) and
should be TDD'd with the DEPTH test written first (RED on current code).
