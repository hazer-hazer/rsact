## 5.14 combinators verification (Task 5.14, 2026-07-13)

`combinators.rs` (`rsact-ui/src/widget/combinators.rs`) is VERIFY-ONLY — no `UnitBuilder` was introduced, and none is warranted. The only live (non-commented-out) type in the file is `Unit`, a fieldless ZST (`pub struct Unit;` — no generics, no fields at all), plus a foreign-type impl `impl<W: WidgetCtx, V: View<W>> View<W> for Option<V>` (derive can't target a foreign/blanket impl anyway). Everything else in the file (`Maybe`, `Conditional`, the `From<F> for El<W>` blanket, `Widget<W> for MaybeReactive<E>`) is commented-out speculative design, not live code — nothing to convert.

Checked whether `Unit` even *could* take `#[derive(View)]`/`#[derive(Builder)]`: it structurally cannot. Both derives' `ctx_param()` helper (`rsact-macros/src/lib.rs`) finds the `WidgetCtx`-bounded generic **on the struct's own generics list** and emits exactly one `impl<...> Trait<W> for Name<...>` tied to it. `Unit` deliberately carries no generics at all — its hand-written `impl<W: WidgetCtx> Widget<W> for Unit` (and the matching hand-written `Build<W>`) is a blanket that lets one non-generic ZST implement `Widget`/`Build` for *every* `W: WidgetCtx` simultaneously (used as the universal placeholder in `Option<V>::into_el`'s `unwrap_or_else(|| Unit.el())`, and anywhere else a "nothing" leaf is needed regardless of which `WidgetCtx` the surrounding tree uses). A derived impl is fundamentally the wrong shape for this: it would tie `Unit` to one specific `W`, which is exactly the property `Unit` exists to avoid. The file already carries a Note-comment recording this reasoning (`combinators.rs`, above the manual `Build` impl: "`Unit: Widget<W>` for *every* `W`, so these need `W` pinned explicitly (the derived identity `Build` on `Foo<W>` doesn't, as it implements `Widget<W>` only for its own `W`)") — left untouched per AGENTS.md.

No build-only props exist to hoist either way: `Unit::build` is a no-op (`let _ = ctx;`), and there are zero fields to partition into build-only vs. runtime. Verified via the existing regression test `el::build::tests::build_transforms_unbuilt_to_built` (`rsact-ui/src/el/build.rs`), which drives `Unit.el()` through the real `BuildCtx::run` arena-build path on `NullWtf` and asserts the node ends up `Built` with a reachable widget — this test passed unchanged before and after the 5.13 Select split (`cargo test -p rsact-ui --lib --features "std,embedded-graphics" -- --test-threads=1`, 77/0). No NEEDS_CONTEXT: `combinators.rs` needs no changes.

## 5.15 icon.rs: repaired, not deferred (Task 5.15, 2026-07-13)

The WS4 status note said icon.rs's repair "needs `SignalOnWrite` — its own pass." That turned out to describe a *design improvement noted in a TODO comment*, not the reason the file failed to compile — none of the 5 compile errors under `cargo check -p rsact-ui --features "std,embedded-graphics,tiny-icons"` trace to reactive-primitive gaps. All 5 were contained, mechanical bugs, fixable with existing `rsact-reactive` primitives (option (b) of the brief), so no `docs/plans` deferral entry was needed for this one — recorded here anyway for the fleet-task log's continuity:

1. **E0061** — `Icon::new` built `Self { value: IconValue::Relative(icon), .. }` (1 arg) against a 2-arg tuple variant (`Relative(Signal<FontSize>, MaybeReactive<I>)`), discarding an already-correctly-built `value` local one line above. A leftover half-edit; fix was using the local.
2. **E0053** — `render(&self, ctx: &mut RenderCtx<'_, W>)` predates the by-value `RenderCtx` signature every other widget in the fleet uses (`fn render(&self, ctx: RenderCtx<'_, W>)` on the trait).
3. **E0310** — the `impl<.., R: ReactivityMarker> Widget<W> for Icon<..>` block needed `R: ReactivityMarker + 'static` to satisfy `Widget<W>: Any`.
4. **E0592** — `IconStyle`'s two `color`-kind fields (`background`, `color`) both defaulted to a method literally named `transparent` (the `declare_widget_style!` macro's default per-field name), colliding. No other widget in the fleet has two unnamed color fields, so this edge case had never been hit; fixed with the same explicit-name-override syntax `label.rs` already uses for `text_color`.
5. **E0282** (+ a type mismatch it was masking) — `ctx.get_style(|t| t.icon, ...)` called a pre-refactor 2-arg `get_style` API (a manual theme-field closure), and `Icon`'s `style` field hand-rolled a 1-arg closure type instead of `WidgetStyleFn<S>`. Deeper cause: `IconStyle` had never been wired into the `Stylist`/`InternalStylist` plumbing at all — no `impl Stylist<IconStyle<C>>` existed for `Theme<C>` or `BinaryTheme`. Fixed the call/field type and added the two impls, gated behind `#[cfg(feature = "tiny-icons")]` (including a duplicated `InternalStylist` trait definition — Rust does not allow `#[cfg]` on individual `+`-joined supertrait terms, so the two arms, with/without `Stylist<IconStyle<C>>`, are two full trait bodies).

None of the errors trace into `rsact-tiny-icons`' generated code (findings §9a.2's SVG-absence caveat does not apply here — `icon.rs`'s errors were entirely in `rsact-ui`).

Repair commit `7cfbc38` (`WS13.4: repair icon.rs (WS4.5 debt)`); split commit `275e6e6` (`WS13.4: split Icon into IconBuilder + retained Icon`) follows the Task 4 recipe — `is_reactive: PhantomData<R>` is the only build-only field, and since `R` is a compile-time-only builder tag with no runtime meaning (a ZST), the shape test is the label/space fallback form (assert `IconBuilder` exists + page builds/renders), not a `size_of` `<` lock. Suites: `cargo test -p rsact-macros` 6/6; `cargo test -p rsact-ui --lib --features "std,embedded-graphics"` 77/0; `cargo test -p rsact-ui --lib --features "std,embedded-graphics,tiny-icons"` 78/0 (+1: `icon_split_builder_exists_and_page_renders`).

## 5.16 image.rs: stays disabled (Task 5.16, 2026-07-13)

`image.rs` (`rsact-ui/src/widget/image.rs`) is `// #[cfg(feature = "embedded-graphics")] // pub mod image;` — commented out of `widget/mod.rs` entirely (not merely feature-gated off, unlike `icon`). Trial-enabling the module (`cargo check -p rsact-ui --features "std,embedded-graphics"`, reverted after) surfaces **10 errors** spanning four independent kinds of drift, confirming revival is nowhere near the "~20 mechanical lines" bar for taking anything but the default:

1. **A deleted type.** `use crate::widget::{MetaTree, ..}` — `MetaTree` no longer exists anywhere in the crate except comments (`grep -rn MetaTree rsact-ui/src` hits only `widget/mod.rs`'s commented-out `// fn meta(&self, id: ElId) -> MetaTree;`, `el/mod.rs`'s two commented-out call sites, and `image.rs` itself). The `meta`/`MetaTree` concept was removed from the `Widget` trait design wholesale (E0432 unresolved import + E0407 "method `meta` is not a member of trait `Widget`").
2. **Missing/never-resolved imports.** `ImageRaw`, `RawDataSlice`, `PixelColor` (E0425/E0405, "cannot find type/trait in this scope") — the file never had a `use embedded_graphics::...` for any of these; whatever glob re-exported them when the file was written no longer does (or never did within this file's own visible history).
3. **A module-privacy change.** `use super::ctx::EventCtx` — E0603, `EventCtx` is a private import at that path now (every other widget reaches it via `widget::prelude::*`, which `image.rs` doesn't use).
4. **A stale trait signature**, same class as icon.rs's E0053: `render(&self, ctx: &mut crate::widget::RenderCtx<'_, W>)` vs. the current by-value `RenderCtx<'_, W>`.
5. **A render-path port, not a mechanical fix** (corrected after review round 1 — an earlier revision of this section wrongly claimed `rsact-render::Renderer` "has no bitmap-blit primitive"; it does). The E0599 itself is real: as written, `embedded_graphics::image::Image::new(&self.data, eg_top_left).draw(ctx.renderer)` requires `ctx.renderer` (an abstract `&mut W::Renderer: rsact_render::renderer::Renderer`) to be an embedded-graphics `DrawTarget`, which it is not — `Renderer` is the backend-agnostic abstraction (`eg`/`tiny_skia` are two of its impls) that the "decouple from embedded-graphics" direction (EVOLUTION.md) put between widgets and the concrete backend. But the abstraction already covers images: `Renderer::image(&mut self, image: DrawImage<'a, Self::Color>) -> RenderResult` exists (`rsact-render/src/renderer.rs:151`) and is implemented in BOTH backends (`eg/renderer.rs:448`/`:609`, `tiny_skia/mod.rs:363`) plus the null renderer's no-op (`renderer.rs:327`), with the supporting types in `rsact-render/src/image/mod.rs` (`ImageRef<'a, C>`/`ImageOwned<C>`/`DrawImage<'a, C>`). So no `rsact-render` design work is needed — what remains is porting the *widget*: replace its data model (`data: ImageRaw<'a, W::Color, BO>` plus the eg-specific `BO: ByteOrder` type param and `PixelColor`/`RawDataSlice` where-clauses) with `rsact-render`'s `ImageRef`, and rewrite `render` to build a `DrawImage` and call `ctx.renderer.image(...)`. That would drop the `#[cfg(feature = "embedded-graphics")]` coupling entirely (the widget would work on tiny-skia too), but it is a rewrite of the widget's storage type, generics, constructor, and draw call — well past the "~20 mechanical lines" bar.

Decision: **stays disabled**, per the brief's default. No code changes made (the trial `pub mod image;` enable was reverted; `git diff --stat rsact-ui/src/widget/mod.rs` is empty). What reviving it for real would take: (a) resolve or replace the `meta`/`MetaTree` references (delete — the trait no longer has the method), (b) drop the eg-specific imports/generics as part of the port in (e), (c) fix the `EventCtx` import path, (d) fix the `render` signature (mechanical, matches icon.rs's fix), and (e) port the data model + render path from direct embedded-graphics types/draw calls onto the existing `rsact_render::image::{ImageRef, DrawImage}` + `Renderer::image` API — the substantive piece, and the natural moment to also do the `ImageBuilder`/`Image` split. The split itself is not the hard part — the render-path port is, and doing (a)–(d) without (e) would just enshrine the eg coupling the codebase is moving away from.
