# WS4.4 — Investigation: zero-source render probes — demote, not dispose

**Date:** 2026-07-09 · **Status:** investigation only — **no mechanism landed** (per the WS4 charter: "do NOT land a demotion mechanism without resolving the conditional-first-run hazard"). · **Feeds:** a future WS4.4/WS6 execution item.

## The opportunity

`rsact-ui` mints one `Probe` per widget **part** it draws (`ElState.part_probes`, WS2). A probe is a runtime node (`ValueKind::Probe`) plus its `subscribers`/`sources` edges. Its only job is answering "did any dependency I read last frame change?" For a part whose render closure reads **no** reactive value, that answer is eternally "no" — the node is pure freight:

- One `ValueKind::Probe` slot (permanent RAM — WS4.6: capacity = peak).
- Per-poll `subscribe` + `maybe_update` dispatch (`run_probe`, `runtime.rs`).

A zero-dependency part still needs **force**-rerendering (parent overdraw / damage regions) — but force needs no node: `Probe::poll(force = true, f)` runs `f` unconditionally (`probe.rs:97`, verified by `probe_force_runs_without_change`), and the "did my deps change?" question is meaningless when there are no deps. So the idea (roadmap sketch):

```rust
enum PartGate {
    Static,          // plain render fn — force still works (poll(force=true) needs no node)
    Probe(Probe),    // parts with real reactive deps keep their graph node
}
```

`ElState`'s part entry becomes a `PartGate`; a `Static` part costs zero runtime nodes and zero edge bookkeeping, re-rendering only when the render frame threads `force` down to it.

## How a probe actually tracks (grounded in the code)

`Probe::poll(force, f)` → `Runtime::run_probe`: `is_alive` → `subscribe` → `maybe_update` → if changed‖force: `clear_sources` → `with_observer(f)` → `mark_clean`. Dependencies are recorded **at read time**: any `.get()`/`.with()` inside `f` calls `track()` → `subscribe`, recording a source edge. Edges are **re-tracked every executed poll** — `clear_sources` detaches the old set first, so a conditionally-dropped dependency is unsubscribed (`probe.rs::probe_conditional_dep_unsubscribed`). "Zero dependencies" therefore means: after `f` ran, this probe's `sources` set is empty.

## The soundness hazard: conditional first runs

"Zero sources on the runs observed so far" is **not** the same as "zero sources forever." A part's dependency set is data-dependent:

```rust
// render closure for a part:
if state.expanded { self.detail_signal.get(); }   // reads a signal only when expanded
```

On run #1 with `expanded == false`, the closure reads nothing → the probe has **empty sources**. If we conclude "static" and demote it to `PartGate::Static` (drop the node), then later, when `expanded` flips to `true`, the closure *does* read `detail_signal` — but there is no probe to record the edge, so a subsequent `detail_signal.set(...)` will **not** re-render the part. Result: **stale UI**, silently. This is the exact dual of the `clear_sources` machinery (which correctly *removes* a dropped dep); a demotion must not treat "no dep this frame" as "no dep ever."

## Options evaluated

**(a) Opt-in: the widget author declares a part static.** `PartGate::Static` is chosen by the widget when it *knows* a part reads nothing reactive (e.g. a fixed divider, a constant-glyph icon). Default posture stays `Probe`. **Sound** (the author has whole-source knowledge the runtime lacks), **zero magic**, zero per-frame cost. Downside: manual, and a part that "looks static" but has a rare conditional read is an author bug — same class as any incorrect manual annotation, but it fails toward stale UI, so it must be used only for provably-constant parts.

**(b) Demote after N clean runs (heuristic).** Observe a part; if its sources have been empty for N consecutive polls, demote. **Rejected — unsound.** N-clean-runs is not evidence of zero-dep-forever: the `expanded` example is clean for arbitrarily many frames before the branch is taken. Any finite N is defeated by a late conditional read. (Recorded as rejected in the roadmap; this investigation confirms it.)

**(c) Lazy re-promotion via a sentinel.** A "static" part is not fully off the graph: it runs under a lightweight **sentinel observer** that mints a real `Probe` and re-registers the moment the closure calls `track()` (i.e. reads its first reactive value). **Sound and automatic** — a part that becomes reactive self-heals on the very frame it first reads a signal, before any write it would miss. Cost: one branch on the hot `track()`/`subscribe` path (is the current observer a promotable sentinel?), and the sentinel still occupies *something* until it either promotes or is proven static. This reclaims the node for genuinely-static parts without author annotation, at the price of complicating the core read path — which the roadmap's reactive-ops-must-stay-small posture (see G7) treats cautiously.

## Recommendation (not landed)

1. **Ship (a) opt-in first** when this item executes. It is provably sound, needs no core-runtime change (it lives entirely in `ElState`/`render_part`: a `PartGate::Static` variant whose render fn is called with the threaded `force`), and matches the framework's "explicit over magic" bias. The default remains `Probe`, so nothing regresses; a widget opts a part in only when it is constant by construction.
2. **Hold (c) lazy re-promotion as the automatic evolution**, gated on evidence: land it only if WS4.6 capacity metrics show a material count of parts left conservatively as probes *and* the one-branch cost on the `track()` path measures negligibly in the reactivity bench. It touches the reactive core, so it wants its own decision gate (adjacent to G7's CS/track-path work).
3. **Never (b).**

## Interactions to respect at execution time

- **4.3 force_redraw demotion.** When `force_redraw` becomes an imperative flag threaded as the `force` argument (WS4.3/WS5), a `PartGate::Static` part is re-rendered exactly by that threaded `force` — the two designs are complementary: static parts have *only* the force path, which is precisely what the imperative flag provides. Sequence 4.4 after (or with) the force-path rework so `Static` parts inherit a single, well-defined force channel rather than the current per-part `force_redraw.track()` edge.
- **WS2 ownership.** Part probes are owned by `ElState` and disposed on `remove_subtree`/`Page::drop` (untracked, WS2). `PartGate::Static` simply stores no probe for that entry; `PartGate::Probe` keeps today's disposal. No new lifecycle.
- **Measurement.** The canonical `ui_labels` parts are **not** zero-dep (a label part reads its reactive `content` memo), so they would stay `PartGate::Probe`; the win lands on genuinely static parts (a `Label` over `"literal".inert()` content, fixed dividers, constant icons). Add a "static-part count" to the 0.4 scenario set when this executes, so the payoff is measured, not assumed.

## Verdict

The demotion is worth doing, but only via **(a) opt-in** as the sound, low-risk mechanism, with **(c)** as an evidence-gated follow-up behind its own gate. The conditional-first-run hazard is real and defeats the tempting heuristic **(b)**; it is resolved (not worked around) by (a) using author knowledge and by (c) self-healing on first `track()`. **No code lands from this investigation.**
