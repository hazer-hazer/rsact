## 2. The five-axis measurement

Pre-split baseline `5fb2ff9`; post-split `4fdfd44`/HEAD. Host `aarch64-apple-darwin`; thumb `.text`/`.rodata` from the `size-probe` bins (`thumbv7m`/`thumbv6m`).

### Axis 1 — struct `size_of` (retained widget vs its builder), HEAD

| | retained | builder | delta |
|---|---|---|---|
| Button | **48 B** | 152 B | −104 B (−68%) |
| Flex | **12 B** | 40 B | −28 B (−70%) |

The retained widget is 68–70% smaller than its builder. **Caveat:** this is a stack `size_of` for a *short-lived* value — the builder is consumed and dropped during `build`; it never resides in the arena. The number that matters for embedded RAM is the arena/heap resident cost (Axis 2).

### Axis 2 — heap RAM per page (the only pre/post-comparable scenario is `ui_labels`)

| scenario | pre heap_live | post heap_live | delta | build_allocs |
|---|---|---|---|---|
| ui_labels_5 | 12 267 B | 11 823 B | **−444 B (−3.6%)** | 65 → 66 (+1) |
| ui_labels_10 | 21 590 B | 20 730 B | **−860 B (−4.0%)** | 99 → 100 (+1) |

A ~4% heap reduction even though `ui_labels` has only **one** split widget (the page-root `Flex`) — that single Flex dropping its `children`-Vec husk. `build_allocs` +1 (the boxed `Build` object in the transform path). Net: neutral-to-slight-win, as spec §6 predicted. (New button/flex-heavy scenarios recorded as HEAD-only reference points; no pre-split baseline exists for them.)

### Axis 3 — reactive node counts (`ui_labels`, pre/post)

**Zero delta** — `ui_labels_5` = 29 nodes pre and post; `ui_labels_10` = 49 both. Exactly the honest expectation: reactive props still bind through the `Layout` node in the prototype, so the node-count win belongs to WS5.1 (node removal), which this split *enables* but does not itself perform.

### Axis 4 — flash `.text`/`.rodata` (thumb), pre vs post

| bin / target | .text delta | .rodata delta |
|---|---|---|
| reactive (any) | **0** | **0** (UI-only split, as expected) |
| ui / thumbv7m | +212 B (+0.34%) | +212 B (+5.78%) |
| ui / thumbv6m | +188 B (+0.28%) | +216 B (+5.76%) |

A small, real uptick (~0.3% `.text`, low-hundreds of bytes) from the shared split infrastructure (Build trait, two-stage node, the two derives). **Caveat:** the `size-probe` `ui` bin only instantiates `Flex::col(Label…)` — **Button is not exercised**, so this reflects the shared infra + the Flex half only; Button's per-widget code is LTO-eliminated here. Not severe enough to threaten the Blue Pill floor (~48 K flash budget). This is the R1 cost the gate must weigh.

### Axis 5 — ergonomics

- **LOC:** button.rs 145→117, flex.rs 224→208 = **−44 LOC** across the two widgets.
- **Per-widget boilerplate:** ~**4.8:1** (58 hand-written `View`+`SingleViewMarker`+`Build` lines → 12 annotation lines).
- **One-time macro cost:** +209 lines (`rsact-macros`), amortized. **Break-even ≈ 9 widgets** vs a ~15-widget fleet → profitable for the full migration.
- **Call-site churn:** 7 `.el()`→`.into_el()` conversions repo-wide (split builders aren't `Widget`, so `.el()` — a `Widget` method — becomes the `View` method `.into_el()`) + ~4 signature disambiguations. Small and mechanical.

## 3. The honest reading

WS13's thesis (analysis §1/§5) holds under measurement: **the split is an enabler, not a RAM win in itself.** In the prototype:
- **Reactive nodes: 0 delta** — the node-count win is WS5.1's (this split makes the clean node-free storage *possible*).
- **Heap: ~4% better** — from dropping the child husks; real but modest.
- **Flash: ~+0.3% `.text`** — a small, acceptable cost for the protocol + derive.
- **Struct size: −68–70%** on the retained widget, but on a transient value.
- **Ergonomics: 4.8:1 boilerplate cut, break-even at ~9 widgets** — the derive makes the fleet migration cheaper per-widget than the status quo.

A near-neutral RAM/flash result is a **success** by the spec's own framing: it confirms the split does not regress the embedded floor while delivering the transform protocol WS5.1 needs and an ergonomic derive. The strategic payoff is unchanged — WS5.1 can now put layouts arena-owned/`ElId`-identified/node-free/`Rc`-free, because build-only props will have a transient home (the builder) instead of inflating the retained widget.

## 4. Deviations discovered during execution (protocol: report)

1. **`LayoutWidget<W>: Widget<W>` supertrait had to be dropped** (Task 2). The spec assumed the builder traits (`SizedWidget`/`BlockModelWidget`/`FontSettingWidget`) could move onto a non-`Widget` builder, but they were bounded on `Widget<W>` via `LayoutWidget`. Fixed by dropping the supertrait and gating the one `Widget`-dependent default method (`FontSettingWidget::font_props`, which is uncalled anywhere) with `where Self: Widget<W>`. Verified a pure loosening — all builder-trait uses are concrete impls; no generic code depended on the supertrait. Foundational for both split widgets.
2. **Retained `Flex<W>` needs a `ctx: PhantomData<W>`** (Task 3). `Flex` holds only `layout: Layout`, leaving `W` unused (E0392). Added a `PhantomData<W>` (matching the `space.rs` precedent). The `Dir` *type param* is still removed (7.2 win); the `children` Vec is still dropped (RAM win). For the derive's by-name move to construct it, `FlexBuilder` carries a matching `#[widget] ctx: PhantomData<W>` (a ZST).
3. **`axis` is a `new` parameter, not a stored field** (Task 3). Storing it produced a dead-code warning; it is only needed to build the layout in `new`.
4. **Latent split breakage in a workspace-excluded consumer** (Task 5). `size-probe/src/bin/ui.rs` used `Flex::col(…).el()`, which stopped compiling after the split (`FlexBuilder` isn't `Widget`). Invisible to `cargo test --workspace` because `size-probe` is excluded. Fixed (`.el()`→`.into_el()`). **Implication for 13.4:** the fleet conversion must sweep excluded crates + examples, not just the workspace.

### Flagged discrepancies (unchanged from the spec, confirmed untouched)
- **`icon.rs` does not compile** — known WS4.5 debt (feature-gated off; not in the prototype's `std,embedded-graphics` build). Untouched. The fleet must land the icon repair before/with converting it.
- **`image.rs` disabled** (pre-`build` API). Untouched. The fleet must explicitly decide revive-vs-exclude.
- **Examples** (`sandbox`, `flex`, `widget_gallery`, `scrollable`, …) are broadly pre-broken on this branch for unrelated reasons (`u8g2_fonts`, private modules, `tiny-icons`) independent of the split.

## 6. Recommendation

**Recommend GO for the fleet (13.4), conditioned on the derive hardening (M1–M3) landing first.** Rationale:
- The protocol is proven correct (token-level oracle-verified derive; 65/0; two-verdict reviews all Approved) and does not regress the embedded floor (flash +0.3%, heap −4%).
- The ergonomics clear the bar decisively: 4.8:1 per-widget boilerplate reduction, break-even at ~9 of ~15 fleet widgets.
- The strategic goal — enabling WS5.1's clean node-free/`Rc`-free/`ElId`-identified layout — is delivered: reactive-node count is unchanged now, and the builder gives build-only props a transient home for when WS5.1 removes the node.

**Before 13.4:** harden the derive (M1 guard; M2 `#[layout(delegate)]` for `Show`; M3 any additional child modes the fleet needs), sweep excluded crates + examples (per §4.4), and land the `icon.rs` repair. **13.4 must not begin until this gate is signed off.**

### WS5.1 hand-off
WS5.1 inherits this protocol. When it removes the `Layout` node, reactive **prop** bindings lose their builder-time residence and route through `Build::build` (which has `ctx.id`) — the same transform this prototype established, extended from children to props. `Dynamic` already proves build-time effect wiring (`dynamic.rs`). At that point the reactive-node count drops (the win Axis 3 shows as 0 today), and the retained widget can shed `layout: Layout` too.

### Also ready to tick on sign-off (folded WS7 items)
7.6 (Widget method set — `build` demoted to a no-op, to be deleted post-fleet), 7.7 (`derive(View)`/`Builder` ctx-param detection), and the 7.2 slice (`Flex` `Dir`→`Axis`) are delivered for the two prototype widgets; the 7.2 remainder (Select/Scrollable/Space/Bar/Slider + `V→numeric`) rides 13.4.
