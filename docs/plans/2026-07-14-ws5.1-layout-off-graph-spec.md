# WS5.1 — Layout off the graph: arena-owned, `ElId`-identified, node-free

**Date:** 2026-07-14 · **Branch (target):** `ws5-incremental-layout` · **Status:** DESIGN SPEC — awaiting maintainer sign-off before implementation.
**Reads with:** the roadmap `docs/plans/2026-07-05-rsact-evolution-roadmap.md` (WS5 §, gate G8), the WS4.0 Layout design-input `docs/plans/2026-07-09-ws4.0-inline-inert-analysis.md` (§3, §100-107), the WS13 residence proof `docs/plans/2026-07-09-ws13-views-as-builders-analysis.md` (§2), and the WS13.1 build-transform spec `docs/plans/2026-07-09-ws13.1-builder-widget-split-spec.md` (§2.7 — the explicit WS5.1 hand-off).

This is the design deliverable the roadmap's 5.1 item and the maintainer's "reinvestigate the layouts case" redirect require before code lands. It answers: **(1)** what "off the graph" concretely means now that WS13 shipped the build-transform protocol; **(2)** where `LayoutData` resides and how the layout tree is walked (the one real architectural fork — §2.3, put to the maintainer in §7); **(3)** the binding-effect + page dirty-set mechanism that replaces the whole-page `Memo<LayoutModel>`; **(4)** the blast radius + migration order, keeping full relayout the default compiled path; **(5)** the TDD plan + acceptance; **(6)** open decisions.

---

## 0. Scope & non-goals

**In scope (5.1):**
- Retire `Layout = Static(ValueId) | Reactive(Signal<LayoutData>)` (the fake-inert node the maintainer objects to, `layout/node.rs:18`). `LayoutData` becomes **owned by the arena, identified by `ElId`** — zero reactive-runtime nodes for layout.
- Reactive layout **props** (`.width(sig)`, `.padding(sig)`, …) bind via a **build-time effect** (created in `Build::build`, which has `ctx.id`) that writes the arena-owned `LayoutData` and **marks the node's `ElId` in a page-level dirty set** — the D3 write→node + D7 pay-per-binding mechanism from one source.
- Replace the whole-page `Memo<LayoutModel>` (`page/mod.rs`) with **relayout gated on the dirty set** (non-empty ⇒ relayout at the next pull point: event hit-test / render). **Full relayout stays the default compiled path**; the dirty set's *contents* are consumed by 5.2 (incremental) and 5.3 (changed-set) later.
- Fold the **memo-entangled 5.0b singletons** (`force_redraw`, `fonts`, `page_style`) — they only exist because the layout memo tracks/writes them; the dirty-set rework dissolves that coupling. (`renderer`/`dev_tools` → `Rc<RefCell>` stay free-standing 5.0b, out of this spec.)
- `Widget::layout`/`Build::layout` return type and the `Copy`/`&Layout` ripple (WS4.0 §105); retire the `layout_mut`/`now_reactive`/`setter` reactive-on-write machinery and the `LayoutMut` trap class.
- `transparent_layout` (Dynamic) maps to the parent explicitly.

**Non-goals (later stages, do not build here):**
- **5.2** retained per-node `(last_inputs, outer_size, min_size, flags)` + skip-clean-subtrees + the differential fuzz test (feature `incremental-layout`). 5.1 delivers the dirty set; 5.2 *consumes* it to make relayout incremental.
- **5.3** changed-set (old∪new rects) output for WS6.
- **5.4** persistent text-measure cache.
- **5.x** the WS1.3b pull-side effect enqueue (rides the 5.2 lazier-marking work).
- `renderer`/`dev_tools` → `Rc<RefCell>` (independent 5.0b, render-poll-closure surgery).

**Acceptance gate for 5.1 (not the full-WS5 acceptance):** the fake-inert is gone (node count for a static page drops by one-per-widget — re-baseline `node_count_regression`); reactive layout props still react (the `reactive_*_setter_persists_and_reacts` guardrails stay green, ported to the new mechanism); a static page does **zero** relayout on an idle frame; existing layout/UI suites green; full relayout remains the default (no `incremental-layout` feature yet). The visit/measure *counts* are unchanged by 5.1 (they are 5.2's target) — 5.1 changes *where the data lives and what triggers relayout*, not the kernel's per-pass work.

## 1. The current shape (post-WS13, on `ws5-incremental-layout` at `7153934`)

Three facts define the problem 5.1 removes.

### 1a. `Layout` is a `Copy` handle into the reactive runtime — the fake-inert
`Layout` (`layout/node.rs:18`) = `Static(ValueId) | Reactive(Signal<LayoutData>)`. `Static` holds a **fake-inert** node: `Layout::inert` calls `rt.create_inert(layout)` and then *writes to it* (`node.rs:40-46`, the doc-comment admits "fake Inert as we write to it"). `now_reactive` (`node.rs:48-69`) upgrades `Static→Reactive` by minting a `Signal`, cloning the data across, and **disposing the old inert** — changing the `ValueId`. `Copy` is load-bearing and dangerous:
- **Load-bearing:** a child widget keeps its own `Layout` field *and* hands a `Copy` of the same handle to its parent's child-collection; the layout pass reads through the **parent-held** copy while builder setters mutate through the **widget-held** copy. They agree only because `Copy` preserves the `ValueId` (WS4.0 §3).
- **Dangerous:** `now_reactive` done through a throwaway `Copy` (`self.layout()` by value) disposes the inert but loses the reassignment → the `LayoutMut` trap, contained by the `layout_mut` supertrait + a dedicated regression (`page/mod.rs:1811-1863`, `EVOLUTION.md:45`).

### 1b. Reactive props bind at *builder* time through the node
`.width(sig)` → `layout_mut().setter(sig.maybe_reactive(), …)` → for the reactive arm, `Layout::now_reactive()` upgrades to a `Signal` and the setter creates a binding **effect immediately** (`widget/mod.rs`, `node.rs:108-129`). This is only possible because the node *is* the residence — it exists at builder time. WS13's §2.7 states plainly: **the node still exists in the prototype specifically because removing it is WS5.1's job.**

### 1c. Three parallel encodings of the child tree
Per analysis §3, children are encoded **three times**: (i) the arena `children` `SecondaryMap<ElId, Vec<ElId>>`; (ii) each widget's build-only child field (`Button.content`, dropped by WS13; `Flex.children` husk); (iii) `FlexLayout.children: MaybeReactive<Vec<Layout>>` / `ContainerLayout.content: Layout` / `ScrollableLayout.content: Layout` — the **layout-handle sub-tree** the kernel actually walks (`model_flex` reads `flex_layout.children`, `model_layout` derefs `*content`). (i) and (iii) are kept structurally parallel by the **WS3.5 positional zip** (`el/mod.rs` `check_children_parallel`, degrade-don't-panic) — a load-bearing invariant precisely *because* identity is implicit. The page-wide `Memo<LayoutModel>` (`page/mod.rs`) tracks every reactive layout signal and recomputes the whole tree on any change, and writes `force_redraw` as a side effect inside the memo.

The kernel itself (`model_layout`/`model_flex`) is **arena-agnostic**: it takes a self-contained `Layout` handle and recurses through the (iii) handle sub-tree.

## 2. The target model

### 2.1 `LayoutData` is arena-owned, keyed by `ElId` — no runtime node
The `LayoutData` for element `id` lives in the arena node for `id` (a `layout: LayoutData` field on `ElData`/`ElState`, or an arena `SecondaryMap<ElId, LayoutData>` — an implementation choice, §7-D). There is **no `ValueId`, no `create_inert`, no `Signal<LayoutData>`** for layout. A static widget mints **zero** reactive nodes for its layout. `ElId` is the identity — "the Layout–El relation is meaningful" (maintainer), so no `ValueId` indirection.

`Layout` the *type* either (a) collapses into "the `LayoutData` at `arena[id]`" (no handle type at all — the accessor is `arena.layout(id)`), or (b) becomes a thin newtype `Layout(ElId)` that indexes the arena. **Recommend (a)**: the handle existed only to carry the `ValueId`; once identity is `ElId` and `ElId` is already the arena key, a separate handle is redundant. `Widget::layout(&self)`/`Build::layout(&self)` then return the widget's **owned initial `LayoutData`** (moved into the arena slot at `arena.add`, exactly like the `{id, layout}` husk stores it today) — not a shared handle. Post-move, all reads/writes go through `arena[id].layout`.

### 2.2 Static props write once; reactive props are build-time effects
- **Static** `.width(300)` / `.padding(4)`: mutate the builder's owned `LayoutData` in place at builder time (as today, but on a plain owned value — no node, no `now_reactive`). It is moved into the arena at build.
- **Reactive** `.width(w_sig)`: the builder **records** the binding (a small `Vec` of closures on the *builder*, its transient home — the analysis §2 residence chain); `Build::build` **drains** them, wiring each as an effect that captures `ctx.id`:
  ```rust
  // Build::build has BuildCtx { arena: Signal<ElArena<W>>, id: ElId } (build.rs:41-44)
  // + a new page_dirty handle. Effect owned by the page scope → survives the move.
  // Wrapped once as ctx.bind_layout(source, set_map) (AGENTS.md: shared wrapper,
  // not copy-pasted per widget) so the write+mark pattern lives in one place:
  ctx.bind_layout(w_sig, |data, w| data.size.set_width((*w).into()));
  // expands to:
  //   let (arena, id, dirty) = (ctx.arena, ctx.id, ctx.page_dirty);
  //   create_effect(move |_| {
  //       let w = w_sig.get();                                    // tracked
  //       arena.update_untracked(|a| set_map(a.layout_mut(id), &w));
  //       dirty.mark(id);                                         // D3 write→node ∧ D7 pay-per-binding
  //   });
  ```
  This is the **identical move `Dynamic` already makes** (`dynamic.rs:55-64`: an effect created in `build` capturing `ctx`, calling `set_single_child`), extended from *structure* to *props* — exactly the WS13.1 §2.7 / WS13.2 §6 hand-off. One effect per reactive binding (pay-per-binding); a fully static widget creates none. **API prerequisite:** `BuildCtx.id`/`.arena` are currently private with no accessor (census §6); 5.1 exposes them via the `bind_layout` wrapper (preferred — keeps `id`/`arena` encapsulated) rather than raw getters. The builder records reactive bindings at builder time (a `Vec` of `FnOnce(&mut BuildCtx)` on the transient builder — its residence per analysis §2) and `Build::build` drains them; the `#[derive(Builder)]` can generate the drain (§7-F).

### 2.3 The layout tree = the arena tree (THE fork — see §7-A)
**Recommended (Option 2, the maintainer's stated vision):** delete the layout-handle sub-tree (`FlexLayout.children`, `ContainerLayout.content`, `ScrollableLayout.content`). The kernel walks the **arena** children: `model_layout` takes `(ctx, arena, id, limits, parent_size)`, reads `arena[id].layout`, and recurses over `arena.children[id]`. Consequences:
- **Collapses the three-way child encoding (§1c) to one** — the arena children map. The widget child-husk Vec and the `FlexLayout.children` handle-vec both disappear; **the WS3.5 positional zip is retired** because identity is now explicit (`ElId`).
- **Unifies reactive structure and reactive props through the arena.** Reactive children (`#[children(reactive)]`, `Dynamic`) already mutate the arena via `set_children`/`set_single_child` build-time effects; walking the arena means the kernel sees structure changes with no second `MaybeReactive<Vec<Layout>>` channel to keep in sync. (Today, reactive children must update *both* the arena and `FlexLayout.children`.)
- **Cost:** the kernel gains an arena dependency (today it is arena-agnostic). Mitigation: pass the arena behind a small `trait LayoutTree { fn data(&self, id) -> &LayoutData; fn children(&self, id) -> &[ElId]; }` the arena implements, keeping `layout/` decoupled from `el/arena` internals. `transparent_layout` (Dynamic) resolves by having the walk skip the transparent node and adopt its single child under the parent (explicit, replacing the `transparent_layout` flag's positional handling).

**Alternative (Option 1, roadmap-literal "make the zip explicit"):** keep the two trees but re-key the handle sub-tree by `ElId` — `FlexLayout.children: Vec<ElId>` etc., kernel looks up `arena[id].layout` per child. Smaller kernel change, but keeps two child encodings and the sync burden (now explicit rather than positional). Does **not** deliver the "collapse the three-way encoding" structural win. Documented in §7-A as the fallback.

### 2.4 The page dirty set replaces the whole-page memo
- A page owns a `dirty: PageDirty` (a `Set<ElId>` behind a `Cell`/`RefCell`, or a `Trigger` + set). Binding effects (§2.2) and reactive-structure effects `mark(id)`.
- The `Memo<LayoutModel>` is **removed**. Relayout is **pulled** at the same points as today (event hit-test, render) and runs **iff the dirty set is non-empty**. **5.1 keeps this a FULL relayout** over the whole arena from the root (the current kernel, now arena-walked); it just no longer runs when nothing changed, and no longer relies on `LayoutModel` `PartialEq` to detect no-change. 5.2 replaces "full relayout" with "relayout only the dirty subtrees + stop-rule," reading the same set. 5.3 turns the set into old∪new rects.
- **This is where 5.0b's `force_redraw` dies:** relayout that produced changes signals redraw through the dirty set / an explicit `needs_redraw`, not a memo side-effect. `fonts` becomes plain data + an explicit "fonts changed ⇒ mark whole tree dirty" call (fonts change at startup, not reactively); `page_style` → `MaybeSignal` (no live writer). These three demotions land **with** this rework because they were only reactive to feed the memo.

### 2.5 `LayoutModel` output carries `ElId` (makes the event/render zip explicit too)
The kernel's output `LayoutModel` tree is built from the arena walk, so each node can carry its `ElId`. The event/render passes then index by `ElId` instead of positional-zipping against the arena (the render/event half of retiring WS3.5). **Whether the computed `LayoutModel` is *stored per-`ElId` in the arena* (the maintainer's "store LayoutModel in El") is deferred to 5.2** (retained tree) — 5.1 keeps `LayoutModel` a returned/owned tree, but ElId-tagged so 5.2/WS6 can key on it. (§7-C.)

### 2.6 What the retained widget looks like after 5.1
Per WS13.2 §6, the retained widget **sheds `layout: Layout`** — its initial `LayoutData` is moved into the arena at `arena.add` and it never holds a layout handle again. `Widget::layout` is then only meaningful *pre-build* (`Build::layout` returns the builder's initial `LayoutData` for the husk); post-build, layout is `arena[id].layout`. (This interacts with the `Build::layout`/husk protocol — §3, §7-B.)

## 4. Migration order (commit series; full relayout stays default)

Each step is a reviewable commit; the suite stays green throughout, and **full relayout stays the compiled default** (no `incremental-layout` feature — that is 5.2). The series is ordered so the risky kernel walk lands behind an accessor that can be bisected against the old path.

1. **`BuildCtx` plumbing** — add the page `dirty` handle + the `ctx.bind_layout(source, set_map)` wrapper (§2.2); expose nothing else. No behavior change yet (nothing calls it). Smallest possible first commit; unblocks 4.
2. **Arena owns `LayoutData`** — `ElData` gains `layout: LayoutData` (§7-D); `arena.add` (`arena.rs:249`) moves the builder's owned `LayoutData` in instead of caching a `Layout` handle; add the `LayoutTree` accessor (`fn data(id)`, `fn children(id)`, §7-E). Old `Layout` handle still present in parallel to bisect.
3. **Kernel walks the arena** (Option 2) — port `model_layout`/`model_flex` to `(ctx, tree, id, limits, parent_size)` reading `tree.data(id)` + recursing `tree.children(id)`; **ElId-tag `LayoutModel`**; delete `FlexLayout.children`/`ContainerLayout.content`/`ScrollableLayout.content` and the parent-assembly sites (flex:58/container:38/button:41/scrollable:144/select:219). Behind a diff-test against the old path (§6).
4. **Reactive props → `bind_layout` effects + dirty set** — convert `SizedWidget`/`BlockModelWidget`/`FontSettingWidget` reactive arms; port the `reactive_*_setter_persists_and_reacts` guardrails (`page/mod.rs:1811-1863`) to assert the effect writes the arena slot + marks dirty. Static arms become owned writes; drop `now_reactive`/`Layout::Static`.
5. **Page dirty set replaces the memo** — remove `Memo<LayoutModel>` (`page/mod.rs:55/137-170`); relayout runs iff dirty non-empty at the render/event/pointer/dev-hover pull points; **fold 5.0b** (`force_redraw`→dirty-driven `needs_redraw`, `fonts`→plain data + explicit "fonts changed ⇒ whole-tree dirty", `page_style`→`MaybeSignal`).
6. **Delete the corpse** — remove the `Layout` handle type, `layout_mut`/`setter`/`now_reactive`, the WS3.5 zip (`check_children_parallel` + positional `.zip()` in event/render), and make event/render index `LayoutModel` by `ElId`; retained widgets shed `layout`.
7. **Re-baseline** `node_count_regression` (static layout nodes drop — the per-widget `Layout::Static` count goes to 0) + confirm `layout_counter_baseline` **unchanged** (5.1 is a storage/trigger change, not a per-pass one).

## 6. TDD plan + acceptance

**Keep green (regression guardrails — port, don't weaken):**
- `reactive_*_setter_persists_and_reacts` (`page/mod.rs:1811-1863`) — the "reactive layout prop actually re-lays-out" contract. Re-expressed against the new mechanism: set a signal → its build-time effect ran → `arena[id].layout` changed → the node's `ElId` is in the dirty set → relayout picks it up.
- `flex_view_sequence` (`page/mod.rs:1790`) — reactive children structural parallelism (now identity-linked via arena, not positional).
- `node_count_regression` + `layout_counter_baseline` (metrics-probe) — re-baseline the first (node drop), assert the second **unchanged**.
- Whole suite: rsact-ui 77/0 · reactive 76/0 · render 6/0 · metrics-probe 16/0.

**New tests (write first, per step):**
- **Zero-node static layout** (step 2/4): a fully static widget/page mints **zero** reactive nodes for layout — assert via `current_runtime_profile()` in metrics-probe (the `Layout::Static` count that `node_count_regression` locks at 6/11 drops to 0).
- **Kernel-walk equivalence** (step 3): a differential test — for a corpus of built pages, the arena-walked `model_layout` output equals the old handle-walked output rect-for-rect. (This is the safety net for the biggest change; a lighter cousin of 5.2's fuzz test, scoped to "same tree, no mutation".)
- **Dirty precision** (step 4): setting one widget's reactive `.width()` marks **exactly** that widget's `ElId` dirty (not the whole tree) — the D3/D7 payoff, and the input 5.2/5.3 depend on.
- **Idle = zero relayout** (step 5): after first paint, an idle frame with nothing dirty runs the kernel **zero** times (assert via `layout-counters` visits delta == 0) — today the memo short-circuits via `PartialEq`; 5.1 must match via the empty dirty set.
- **`transparent_layout` maps to parent** (step 3/6): a `Dynamic` child's layout resolves under its parent (no phantom node), verified through the ElId-tagged model.

**5.1 acceptance (restated from §0):** fake-inert gone; static page = 0 layout nodes; reactive props react + mark only their own `ElId`; idle frame = 0 relayout; visit/measure counts unchanged; full relayout still the default; all suites green. (The *incremental* acceptance — 1 visit / ≤2 measures for a Fixed×Fixed label change — is **5.2's**, not asserted here.)

## 7. Open decisions for the maintainer

**DECIDED 2026-07-14 (maintainer):** **A = Option 2** (one tree — arena-walked; delete the layout-handle sub-tree; collapse the three-way child encoding; retire the WS3.5 zip). **B–F = the recommended options**: B = no `Layout` handle type (accessor `arena.layout(id)`); C = `LayoutModel` stays a returned ElId-tagged tree in 5.1, store-in-`El` deferred to 5.2; D = `LayoutData` as a field on `ElData`; E = `LayoutTree` trait indirection; F = `BuildCtx::bind_layout` wrapper + derive-generated drain. Implementation proceeds on this basis (steps 1–3 as PR 1, 4–7 as PR 2).

- **A — the tree fork (§2.3):** Option 2 (one tree = arena; delete the layout-handle sub-tree; kill the three-way child encoding + WS3.5 zip) **[recommended, matches your redirect-#1 vision]** vs Option 1 (two trees, handle sub-tree re-keyed by `ElId`; smaller kernel change; keeps the sync burden). This is the biggest decision and shapes everything downstream.
- **B — `Layout` type fate (§2.1):** collapse to "no handle, accessor is `arena.layout(id)`" **[recommended]** vs a thin `Layout(ElId)` newtype. Affects `Widget::layout`/`Build::layout` signatures.
- **C — `LayoutModel` storage (§2.5):** keep `LayoutModel` a returned ElId-tagged tree in 5.1, store-per-`ElId`-in-`El` in 5.2 **[recommended]** vs pull the store-in-El forward into 5.1.

## 7b. Implementation log — refinements found during coding (2026-07-14)

Recorded as they were discovered so the next session inherits the actual plan, not the pre-code one.

- **Branch:** work is on `ws5.1-layout-off-graph` (stacked on the WS5.0 branch `ws5-incremental-layout` = PR #20). WS5.1 opens its own PR(s); PR #20 stays WS5.0-only.
- **Dirty set lives in `ElArena`, not a separate `page_dirty` `Signal`** (refines §2.4/§7-D/§7-F). A `Signal<DirtySet>` marked/read untracked would re-create the very fake-inert smell WS5.1 removes. The binding effects already capture `ctx.arena`, so `arena.update_untracked(|a| { a.layout_mut(id)…; a.mark_dirty(id); })` needs no extra handle. `bind_layout` therefore captures only `ctx.arena` + `ctx.id`. **Landed:** `el/dirty.rs` (`DirtySet`, tested); the `ElArena` field + `mark_dirty`/`take_dirty` accessors land with the reactive-props stage that first uses them.
- **The port is *in-place*, not parallel — no drift, so the existing suite is the equivalence guarantee.** `LayoutTree::layout(id)` reads the node's *live* `Widget::layout()` handle through the arena, so the arena walk sees exactly today's (reactive-current) data. Changing only the child *source* (arena `effective_children` instead of `FlexLayout.children`) yields a byte-identical `LayoutModel` (same nodes/order/rects), so render/event (still positional-zip) and every existing layout/UI test pass unchanged — that green suite *is* the differential check. (The exhaustive random-tree fuzz stays WS5.2's, as planned.) **`LayoutModel` ElId-tagging is deferred to PR2** (it's only needed to retire the zip); PR1 keeps the positional zip and just re-sources children. **Landed:** `layout/tree.rs` (`LayoutTree` + `effective_children` with transparent flattening, tested).
- **`min_size` must also thread `(tree, id)`.** `LayoutData::min_size`/`FlexLayout::min_size`/`ContainerLayout::min_size`/`ScrollableLayout::min_size` recurse into children via the handle sub-tree (`FlexLayout.children`, `*content`). The kernel port must convert these to walk `tree.children(id)` too; otherwise the structural walk is arena-based but `min_size` still reads the handle tree (a correct-but-hybrid intermediate — both trees are parallel — that leaves the handle tree load-bearing). Doing structural + `min_size` together is what makes `FlexLayout.children`/`*content` fully unread and deletable. This is the bulk of the remaining PR1 kernel work.
- **`ContentLayout::content_sizing` is a leaf** (no children) — untouched by the tree threading; only the *recursive* `min_size` arms change.

## 8. Risk & sequencing note

This is the roadmap's **medium-high risk** stage. The two genuinely hard parts are **step 3** (kernel walks the arena — a structural change to the layout solver, de-risked by the step-3 equivalence diff-test) and **step 5** (removing the page memo without regressing the idle-frame short-circuit or the redraw gating). The `Copy`-removal / `&LayoutData` ripple (steps 4/6) is wide but mechanical (WS4.0 §105 pre-sized it; the compiler finds every site). Because full relayout stays the default and `incremental-layout` (5.2) is not introduced here, 5.1 is shippable and testable on its own — the perf win (visit/measure reduction) arrives with 5.2 consuming the dirty set this stage produces.

**Estimated: 2 sessions** (steps 1-3 = the storage+kernel core; steps 4-7 = the reactive/memo/cleanup). Recommend landing steps 1-3 as one PR (behind the equivalence test, old path still present) and 4-7 as a second, so the kernel change is reviewable in isolation.
