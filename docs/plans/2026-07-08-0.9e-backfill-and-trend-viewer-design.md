# WS0.9e — Historical backfill + per-commit trend charts (design)

Date: 2026-07-08 · Roadmap item: `docs/plans/2026-07-05-rsact-evolution-roadmap.md` WS0 §0.9e

## Goal

Make the GitHub-Pages metrics dashboard chart **every recorded property per commit**
across the whole history the tool can measure, not just the PR/push points that CI
happened to catch. Three parts, all serving that one goal:

- **(A)** a one-time (and forever-repeatable) **backfill/gap-fill** job that records a
  snapshot for every historical commit currently missing one;
- **(B)** an **ordering index** so the browser can put commits on a correct x-axis
  without running git;
- **(C)** an **interactive trend viewer** (table with domain-aware improvement arrows +
  expandable per-row charts + a big multi-series sidepanel chart).

Informational only — like all of 0.9, it never gates a merge (the 0.4 regression test is
the hard gate).

## Boundaries & non-goals

- **Tool-birth hard boundary.** metrics-probe was born at `257f587`. A commit can only be
  measured by instruments that exist *inside it*, so backfill starts at `257f587`. The
  roadmap text cited `f298b98` as the range default — that is **4 commits too early**
  (predates the tool); the job clamps the low end to `257f587`. Commits older than that
  stay covered by the frozen audit rows in the roadmap's Baselines section. *(Roadmap 0.9e
  will be corrected to cite `257f587`.)*
- **`--benches` birth = `f42c154`.** The `--benches` flag/schema arrived with 0.9d. Of the
  28 commits missing a snapshot, only **6** postdate `f42c154`; the other 22 are Layer-1
  only *by construction* (their metrics-probe has no `--benches`). Absent bench medians
  render as gaps, never zeros.
- **CI-runner only.** Heap **bytes** differ between a laptop and the CI runner; one store
  must never mix them. Backfill therefore runs only on the CI runner class via
  `workflow_dispatch` — never local-publish. (The script is still runnable locally for
  *orchestration/idempotency* testing, just not for publishing.)
- **No gating, no thresholds, no alerts** (carried from 0.9d).

## Part A — Backfill / gap-fill (`scripts/ci-backfill.sh` + `workflow_dispatch`)

Per 0.9's "every CI step is also a runnable script" rule, the loop lives in
`scripts/ci-backfill.sh`; a new `backfill` job in `.github/workflows/metrics.yml`
(`workflow_dispatch`, `fetch-depth: 0`, thumb targets installed) just invokes it.

**Inputs:** optional `range` (default `257f587..HEAD`). Low end is clamped to `257f587`
regardless of input.

**Algorithm:**

1. Pull the durable history from `metrics-data` (`snapshots/` + `index.json`) so existing
   revs are known locally.
2. `revs = git rev-list --first-parent <range>` (already ordered newest→oldest).
3. For each `rev` **without** a `snapshots/<rev>.json` (or `<rev>-dirty` — won't occur for
   real commits):
   a. `git worktree add <tmp> <rev>`;
   b. in the worktree, run **that commit's own** tool:
      `cargo run -q -p metrics-probe -- record` plus
      - `--benches` **iff** `git merge-base --is-ancestor f42c154 <rev>` — first run the
        bounded criterion groups (`rm -rf target/criterion` then the commit's own
        `reactivity`+`layout` benches with `--warm-up-time 0.5 --measurement-time 1.0
        --sample-size 10`), best-effort;
      - `--sizes` **iff** the commit is in the sparse subset (**every 5th** first-parent
        commit in range), best-effort;
   c. copy the emitted `metrics/snapshots/<rev>.json` into the accumulation dir; discard
      the worktree's own regenerated `index.html` (it's the *old* viewer);
   d. `git worktree remove --force <tmp>`.
4. **Resilience:** any commit whose probe fails to build/run (known historical bit-rot,
   e.g. the WS2 size-probe) is logged and skipped — never aborts the batch. Count and
   report skipped revs at the end (no silent truncation).
5. **Finalize:** regenerate `index.json` + `index.html` with **HEAD's** metrics-probe (the
   new viewer) over the full accumulated set, then the existing peaceiris step publishes
   `./metrics` to `metrics-data` with `keep_files: true`.

**Idempotency:** the missing-check in (3) is the whole mechanism — a re-run finds nothing
missing and only regenerates the viewer (a no-op to the data). This also means the same job
repairs any future CI-missed push forever.

**Cargo cache:** `Swatinem/rust-cache@v2` keeps per-commit rebuilds cheap; still, wall-time
is unknown until the first real run and will be recorded in the roadmap.

## Part B — Ordering index (`metrics/index.json`)

Snapshots are keyed by hash; hashes don't self-order, and backfilled snapshots all carry
~one `recorded_at` (backfill wall-clock), so the viewer's current `recorded_at` sort is
wrong for history. Introduce a durable ordering artifact.

**Schema** (`metrics/index.json`):

```json
{ "<full_rev>": { "date": <committer_unix_secs>, "parent": "<parent_full_hash>", "branch": "<name-rev hint>" } }
```

**Maintenance — incremental, shallow-checkout-safe.** On every `record`, metrics-probe
merges HEAD's own entry into the pulled `index.json`. All three fields are resolvable at
`fetch-depth: 1`: committer date (`git show -s --format=%ct HEAD`), parent **hash**
(`git rev-parse HEAD^` — the hash lives in HEAD's commit object even when the parent object
isn't fetched), branch hint (`git name-rev --name-only HEAD`). The full-history backfill
fills entries for every commit it touches. `ci-metrics.sh` is updated to also pull
`index.json` from `metrics-data` before recording so the merge is against the live index.

**Ordering in the viewer:** primarily by committer `date` (stored per rev — the real commit
time, present for both push-recorded and backfilled snapshots), tie-broken by ancestor
topology over `parent` edges (transitively skipping revs absent from the set). Date is
primary, not topology, because parent links are **not always available**: a shallow CI
checkout (`fetch-depth: 1`) makes HEAD a parentless boundary commit, so `record` stores
`parent == ""`. Committer date keeps push-recorded and backfilled commits correctly
interleaved regardless; topology only refines genuinely equal timestamps. (Discovered on
the first real push — the incremental entry came back `parent: ""`; see the code doc on
`topo_order`.) Purely informational `branch` hint may annotate columns.

metrics-probe writes `index.json` as a durable file (satisfies the roadmap's "emits an
index.json on metrics-data") **and** inlines the ordering data into `index.html` so the
browser needs no fetch (preserves `file://`).

## Part C — Interactive trend viewer (rewrite of the `html.rs` page)

Still a single self-contained document (all snapshots + index inlined as JSON), **zero
external dependencies**, inline SVG + vanilla JS, opens from `file://`.

**Metric direction (`lowerIsBetter`).** Improvement is domain-aware, not numeric-direction.
Every current metric is **lower-is-better**: node counts (total/signals/memos/effects/
observers/stored), `heap_live_bytes`, `heap_peak_bytes`, `build_allocs`,
`idle_frame_allocs`, `change_frame_allocs`, `layout_visits`, `layout_measures`, section
`.text/.rodata/.bss`, and bench `median_ns`. A central map encodes this (default
lower-is-better; higher-is-better metrics can be added later — none today).

**Main table.** Rows = metrics, columns = commits in **topo order** (from index.json),
newest at the right. Each cell shows the **value** plus an improvement marker vs the
*previous present* commit (gaps skipped): **▲ green = improved**, **▼ red = regressed**,
neutral when equal or no prior. "Improved" = value moved in the better direction, so *fewer
allocs / faster ns / smaller .text* is green ▲ even though the number went down.

**Row expand.** Clicking a row toggles it **selected**: a **compact inline SVG line chart**
appears directly under the row (that metric's **absolute** trend across commits, gaps as
line breaks). Multiple rows can be expanded at once.

**Right sidepanel — big interactive chart.** Overlays **all selected rows**; x = commit
(topo order), y = **absolute value** (never deltas — the arrows carry improvement, charts
carry magnitude). **Multi-series scaling:** a single selected series uses a **true absolute
0-based axis**; when multiple series are selected they are **each normalized to their own
0..max** so differently-scaled shapes (heap bytes vs node counts vs ns) are all visible for
spotting correlations. **Hover → tooltip** listing every selected series' **absolute** value
at that commit. A "select all / clear" control can throw all rows on at once. **Gaps render
as line breaks, never zeros** (the 0.7e/0.7f lesson).

## Data flow

```
backfill job (CI, full history) ─┐
push/master record (CI)          ├─> metrics/snapshots/<rev>.json  ─┐
                                 └─> metrics/index.json (merged)   ─┤
                                                                    ├─> metrics-probe html (HEAD)
                                                                    │     → metrics/index.html (inlines snapshots + index)
                                                                    └─> peaceiris publish → metrics-data → GitHub Pages
```

## Testing & rollout

1. Implement the metrics-probe changes (index.json emit/merge; viewer rewrite).
2. **Local first:** hand-build a multi-snapshot fixture dir (deliberate gaps, one
   improvement, one regression, bench medians present on only some snapshots, a missing
   `--sizes` on most). Run `metrics-probe html` over it, open `index.html` in a browser,
   and verify: topo ordering, per-cell ▲/▼ arrows with correct domain semantics, row
   expand → compact chart, sidepanel overlay with per-series normalization + absolute
   tooltip, gaps as breaks (not zeros). **No CI touched yet.**
3. Add `scripts/ci-backfill.sh`; verify orchestration + idempotency locally (record into a
   scratch dir; do **not** publish laptop bytes).
4. Wire the `workflow_dispatch` `backfill` job into `metrics.yml`.
5. **Trigger the backfill once on GitHub.** Verify: charts populate `257f587..HEAD` with
   visible gaps where instruments didn't yet exist; a **re-run is a no-op**; record the
   **actual CI minutes** in the roadmap.

## Acceptance criteria

- After one backfill run, Pages charts every Layer-1 metric + bench medians from `257f587`
  to HEAD, with gaps (not zeros) where an instrument didn't yet exist.
- Re-running the backfill job changes no data (regenerates the viewer only).
- Table cells show domain-correct ▲/▼ improvement markers; rows expand to inline charts;
  the sidepanel overlays multiple normalized series with an absolute-value hover tooltip.
- `index.json` exists on `metrics-data` and drives commit ordering with no git in the
  browser.
- Nothing gates; no thresholds.

## Risks / open notes

- **CI minutes unknown until the first run** — 28 Layer-1 builds + 6 bench runs + ~7 thumb
  size builds. If prohibitive, the fallback is to drop the sparse `--sizes` (already the
  cheap tier) or split the backfill across dispatches by `range`.
- **Bench backfill is best-effort** — old bench harnesses may not build/rename cleanly; a
  failed bench for a commit just leaves that commit's bench cells as gaps.
- **Force-pushed/rebased-away parents** are unresolvable → those revs fall back to `date`
  ordering (acceptable; they're historical curiosities).
- **Viewer is a full rewrite** of the page template — the existing table semantics (muted
  gaps, dirty markers, bench ±CI framing) must be preserved inside the new structure.
