# WS13.1 — Builder/Widget split: the build-transform protocol (design spec)

**Date:** 2026-07-09 · **Author:** WS13 session · **Status:** design spec (13.1 deliverable) — the concrete protocol + derive + migration, feeding the 13.2 prototype and the 13.3 rollout gate.
**Depends on / builds on:** `docs/plans/2026-07-09-ws13-views-as-builders-analysis.md` (scoping, go/no-go, blast radius — do not re-derive; this spec turns its recommendation §4 into an executable protocol). WS4 landed (PR #11); code re-verified 2026-07-09 against this branch.
**Feeds:** WS5.1 (the clean node-free / `Rc`-free / `ElId`-identified off-graph Layout the split *enables*); folds WS7.2 / 7.6 / 7.7.

---

## 0. Scope & non-goals

**In scope (this spec):** the concrete `build`-transform protocol (how `El`/arena/`build` change), the boilerplate-killing derive, the general per-widget migration recipe, the two prototype conversions (Button, Flex), the design-around cases, and the folded 7.6/7.7/7.2 changes — *as they land on the two prototype widgets*. Full-relayout stays the default path; nothing here rides `incremental-layout` (that is WS5.2).

**Non-goals (explicitly deferred):**
- **Removing the `Layout` node / off-graph layout** — that is WS5.1. This split is the *enabler*; it does **not** itself remove the fake-inert node. Consequence (see §2.7): in the WS13 prototype reactive **props** still bind at *builder time via the node*, exactly as today. Only build-only **child** fields move off the retained widget here. The whole point is to prove the protocol WS5.1 will use to also route prop bindings through `build`.
- **Fleet conversion** of the other ~13 widgets — WS13.4, gated behind the 13.3 maintainer sign-off. This spec ships a protocol under which unconverted widgets keep working **unchanged**.
- **The G4 ctx-collapse (7.5)** — kept `W`-generic; not G4-gated.
- **7.2's `V → numeric` de-genericization** and Dir-removal on `Select`/`Scrollable`/`Space`/`Bar`/`Slider` — fleet work; the prototype de-generics `Flex` only (§4).

---

## 1. The decided shape (recap, from analysis §4 — verify-only)

Shape **(a)/(c)**: an explicit transient **builder** per split widget, consumed at build into a **lean retained widget**; `build` *transforms the type* (builder → widget). The `pending`-queue-on-the-widget half-measure (b) is rejected — it leaves a `Vec` header + heap alloc + construction concerns on the retained type. A derive absorbs the boilerplate.

The pivotal fact the census pinned (analysis §1): **today there is no prop inflation** — reactive props resolve at builder time onto the `Layout` node, leaving zero residue; child widgets are already `mem::replace`'d into the arena, leaving only `{id, layout}` **husks** in parent fields. So the split's *direct* RAM win is modest (drop the child husks + a `PhantomData` type param); its real value is **enabling** node-free WS5.1 and collapsing the three-way child encoding. This spec keeps that honesty front-and-center (§6).

## 2. The build-transform protocol

### 2.1 The `Build<W>` trait (new)

A builder is a construction-only type. It is what `into_el` wraps and what the arena holds **before** the build pass. `build` consumes it (`self: Box<Self>`) into the retained `Box<dyn Widget<W>>`.

```rust
// rsact-ui/src/el/build.rs  (or a new el/build_trait.rs — colocated with BuildCtx)
pub trait Build<W: WidgetCtx>: Any {
    /// Consume the builder into its retained widget, wiring children and
    /// build-time effects with `ctx`. THIS is the type transform (analysis §4c).
    /// `self: Box<Self>` keeps it object-safe (`Box`-receiver dyn dispatch).
    fn build(self: Box<Self>, ctx: BuildCtx<W>) -> Box<dyn Widget<W>>;

    /// Layout snapshot the parent stores in its `El::Stored { id, layout }`
    /// husk (read by `arena.add` BEFORE build; §2.6). For split builders this
    /// returns the builder's own `layout`; for the `Show`-style delegate it
    /// returns the child's layout (design-around, §5.4).
    fn layout(&self) -> Layout;

    /// Read once, pre-build, to seed `ElState` (§2.3). A split builder MUST
    /// report the same flags its retained widget would.
    fn flags(&self) -> WidgetFlags { WidgetFlags::default() }

    fn debug_name(&self) -> &'static str { core::any::type_name::<Self>() }
}
```

Object safety: `fn build(self: Box<Self>, …)` is object-safe (`Box<Self>` receiver). Verified against the existing `Box<dyn Widget<W>>` erasure already in `ElData`.

### 2.2 The trimmed `Widget<W>` trait — the 7.6 fold

`build` **moves off** `Widget` onto `Build`. The retained-widget trait becomes the pure lifecycle set:

```rust
pub trait Widget<W>: Any where W: WidgetCtx {
    fn flags(&self) -> WidgetFlags { WidgetFlags::default() }
    fn debug_name(&self) -> &'static str { core::any::type_name::<Self>() }
    fn layout(&self) -> Layout;
    fn on_event(&mut self, ctx: EventCtx<'_, W>) -> EventResponse;
    fn render(&self, ctx: RenderCtx<'_, W>) -> RenderResult;
    fn update(&mut self, mut ctx: UpdateCtx<'_, W>) -> UpdateResult { ctx.handle() }

    // TRANSITIONAL (bounded prototype only): kept as a defaulted no-op so the
    // ~13 unconverted widgets keep their existing `fn build(&mut self, ctx)`
    // override, which the derived identity `Build` (§2.4) calls. Once the fleet
    // is split (13.4) this method is deleted outright — the final 7.6 shape.
    fn build(&mut self, _ctx: BuildCtx<W>) {}
}
```

Rationale for keeping a transitional defaulted `Widget::build`: it is what makes the prototype *bounded*. Unconverted widgets need **zero** edits — they keep their `Widget::build`; the derive bridges them into the transform-based arena via an identity `Build` that calls it. A hard removal of `Widget::build` would force all ~15 widgets to migrate in one shot, which is exactly the un-bounded rollout the 13.3 gate exists to authorize.

The `el(self)` convenience (`Widget::el`) is **kept** and re-pointed in §2.5 (it has ~25 in-crate call sites; churning them is needless).

### 2.3 `El` / `ElData` / `ElStage` and the arena transform

The arena node holds a **builder before build, a widget after**. Today `ElData.widget: Box<dyn Widget<W>>`; that becomes a two-stage payload:

```rust
// rsact-ui/src/el/mod.rs
pub enum ElStage<W: WidgetCtx> {
    Unbuilt(Box<dyn Build<W>>),   // set by `El::new`; consumed by build_el
    Built(Box<dyn Widget<W>>),    // the retained widget, drives the hot loop
}

pub struct ElData<W: WidgetCtx> {
    pub stage: ElStage<W>,
    pub state: ElState<W>,
}

impl<W: WidgetCtx> ElData<W> {
    pub fn new(builder: Box<dyn Build<W>>) -> Self {
        let state = ElState::for_builder(builder.as_ref());   // reads flags/debug_name
        Self { stage: ElStage::Unbuilt(builder), state }
    }

    /// Post-build accessors. Unbuilt-at-runtime is a bug: log + degrade
    /// (never panic — "UI must never panic" invariant). Returns Option so the
    /// hot-loop callers keep their existing degrade-on-None discipline.
    pub fn widget(&self) -> Option<&dyn Widget<W>> { /* match Built => Some, Unbuilt => {error!; None} */ }
    pub fn widget_mut(&mut self) -> Option<&mut (dyn Widget<W> + 'static)> { /* … */ }
}
```

`El<W>` keeps its two variants; `El::New` now carries the builder-stage `ElData`, and `layout()` reads through `Build::layout` pre-build:

```rust
pub enum El<W: WidgetCtx> {
    New(ElData<W>),                       // Unbuilt inside
    Stored { id: ElId, layout: Layout },  // the husk, unchanged
}

impl<W: WidgetCtx> El<W> {
    pub(crate) fn new(builder: impl Build<W> + 'static) -> Self {
        Self::New(ElData::new(Box::new(builder)))
    }
    pub(crate) fn layout(&self) -> Layout {
        match self {
            Self::New(data) => match &data.stage {
                ElStage::Unbuilt(b) => b.layout(),
                ElStage::Built(w)   => w.layout(),   // defensive; New should be Unbuilt
            },
            Self::Stored { layout, .. } => *layout,
        }
    }
}
```

`BuildCtx::build_el` (`el/build.rs:74-94`) performs the transform in place:

```rust
fn build_el(&mut self, id: ElId) {
    let Some(mut data) = self.arena.update_untracked(|a| a.take_el(id)) else { warn!(…); return };
    if data.state.built {
        error!("Attempt to rebuild element {id:?}");
    } else {
        // take the builder out, run the transform, store the widget
        data.stage = match core::mem::replace(&mut data.stage, /* poisoned sentinel */) {
            ElStage::Unbuilt(builder) => ElStage::Built(builder.build(self.for_el(id))),
            already @ ElStage::Built(_) => { error!("…already built…"); already }
        };
        data.state.built = true;
    }
    self.arena.update_untracked(|a| a.restore_el(id, data));
}
```

(The `mem::replace` sentinel: either add an `ElStage::Poisoned` unit variant used only inside this swap, or destructure the whole `ElData` — an implementation detail for the plan; the invariant is "one transform, top-down, take/restore stays sound" — the same soundness note already in `arena.rs:8-14`.)

**Arena accessor ripple.** Every current `.widget` field access (`ElData` in `arena.rs`, the event/render/update passes) becomes `.widget()` / `.widget_mut()` returning `Option`. `ElState::for_widget` → `ElState::for_builder(&dyn Build)`. The `Box<dyn Widget>` in `remove_subtree`/dispose paths is unaffected (probes live on `ElState`, not the widget). This is the mechanical bulk of the arena change and is prototype-visible even though only two widgets split, because the *node* type changes for all.

### 2.4 Coexistence — identity `Build` for unconverted widgets (NO blanket)

The ~13 unconverted widgets must satisfy `Build<W>` so they can sit in the same arena. A blanket `impl<W, T: Widget<W>> Build<W> for T` is **impossible** — the exact coherence trap already documented for `View` (`el/view.rs:18-22`): a blanket over `Widget` overlaps any concrete `impl Build for FooBuilder` because the compiler cannot prove `FooBuilder: !Widget` (E0119, "upstream may impl", no negative reasoning). So the identity bridge is emitted **per-type by the derive**, never as a blanket:

```rust
// emitted by #[derive(View)] for an UNSPLIT widget Foo (self is both builder & widget):
impl<…> Build<W> for Foo<…> {
    fn build(mut self: Box<Self>, ctx: BuildCtx<W>) -> Box<dyn Widget<W>> {
        Widget::build(&mut *self, ctx);   // existing in-place build (children etc.)
        self                               // identity transform: Foo is its own widget
    }
    fn layout(&self) -> Layout { Widget::layout(&*self) }
    fn flags(&self) -> WidgetFlags { Widget::flags(&*self) }
    fn debug_name(&self) -> &'static str { Widget::debug_name(&*self) }
}
```

Because it is per-type (like the per-widget `View` impls that dodge the same trap), there is no overlap with a split builder's hand/derive-written `impl Build for FooBuilder`. Net: **unconverted widgets are untouched** — same struct, same `Widget` impl, same `fn build(&mut self)`. The derive is the only thing that changes for them, and they already carry `#[derive(View)]`.

### 2.5 `View` / `into_el` / `El::new` re-pointing

`El::new` now takes `impl Build<W>` (was `impl Widget<W>`). `View::into_el` routes through it. The derive emits, for both unsplit widgets and split builders:

```rust
fn into_el(self) -> El<W> { rsact_ui::el::El::new(self) }   // was: Widget::el(self)
```

`Widget::el` (`widget/mod.rs:43-48`) is dropped from the trait (it required `Self: Widget`, which builders are not). Hand-written call sites that used `.el()` switch to `View::into_el` (they are few and in-crate; the plan enumerates them). `El::new` stays `pub(crate)`; `into_el` stays the public conversion.

### 2.6 The husk & arena↔layout parallelism — unchanged

`arena.add` (`arena.rs:247-267`) still reads `el.layout()` (now via `Build::layout`, §2.3) before `mem::replace(el, El::Stored { id, layout })`. The `{id, layout}` husk, the positional arena↔layout zip in the event/render passes, and `check_children_parallel`'s degrade-don't-panic contract (`el/mod.rs:44-59`) are **all unchanged** — WS5.1, not WS13, makes identity explicit and retires the zip. The split neither helps nor harms this invariant; it must simply preserve it (regression-covered by `flex_view_sequence`, `page/mod.rs:1790`).

### 2.7 Where reactive props bind — the honest WS13-vs-WS5.1 boundary

**In the WS13 prototype the `Layout` node still exists**, so a reactive `.width(sig)` on a builder still calls `layout_mut().setter(…)` → `Layout::now_reactive()` and binds the effect **at builder time via the node**, exactly as today (`widget/mod.rs:112-124`, `layout/node.rs`). The builder traits (`SizedWidget`/`BlockModelWidget`/`FontSettingWidget` + `LayoutWidget`) simply **move to the builder type** for split widgets (they stay on the widget for unsplit ones). The retained `Button`/`Flex` therefore *still hold* `layout: Layout` (the node handle) — WS13 does not remove it.

What WS13 removes from the retained widget is only the **build-only child fields** (`Button.content`, `Flex.children`) and Flex's `PhantomData` type param.

WS5.1 is what removes the node; at that point the props lose their builder-time residence and their bindings move into `Build::build` (which has `ctx.id`) — the *same* transform this spec establishes, extended from children to props. `Dynamic` already proves build-time effect wiring works (`dynamic.rs:55-64`: an effect created in `build` capturing `ctx`). **This boundary is the core message to the 13.3 gate:** WS13 buys a small RAM win now and the *protocol* that makes WS5.1's clean storage possible.

## 3. The derive — 7.7 fold + boilerplate killer

### 3.1 Extended `#[derive(View)]` (unsplit widgets)

Today (`rsact-macros/src/lib.rs:45-74`) it emits `View` + `SingleViewMarker` and blindly pushes `'static` onto every type param, hard-coding the literal `W` in `View<W>`. Extended responsibilities:
1. emit `View::into_el` via `El::new(self)` (§2.5);
2. emit `SingleViewMarker`;
3. **emit the identity `Build` impl** (§2.4);
4. robust ctx-param detection (§3.3).

### 3.2 `#[derive(Builder)]` (split builders) + field attributes

New derive for the transient builder type. Declares the retained widget and which fields are children vs moved:

```rust
#[derive(Builder)]
#[builds(Button<W>)]                     // the retained Widget type this consumes into
pub struct ButtonBuilder<W: WidgetCtx> {
    #[widget] layout: Layout,            // moved verbatim into the widget ctor
    #[widget] style: WidgetStyleFn<ButtonStyle<W::Color>>,
    #[widget] on_click: Option<Box<dyn FnMut()>>,
    #[child(single)] content: El<W>,     // ctx.set_single_child(&mut _); dropped
}
```

Generated:
```rust
impl<…> View<W>              for ButtonBuilder<…> { fn into_el(self) -> El<W> { El::new(self) } }
impl<…> SingleViewMarker     for ButtonBuilder<…> {}
impl<…> Build<W>             for ButtonBuilder<…> {
    fn build(self: Box<Self>, mut ctx: BuildCtx<W>) -> Box<dyn Widget<W>> {
        let mut b = *self;
        ctx.set_single_child(&mut b.content);                 // per #[child(single)]
        Box::new(Button { layout: b.layout, style: b.style, on_click: b.on_click })
    }
    fn layout(&self) -> Layout { self.layout }                // or delegate, see #[layout(delegate = …)]
    fn flags(&self) -> WidgetFlags { <Button<W>>::_builder_flags() }  // match the widget
}
```

Child-wiring modes the derive must support (only the first two are needed for the prototype):
- `#[child(single)]` → `ctx.set_single_child(&mut f)` — **Button**.
- `#[children(reactive)]` on a `MaybeSignal<Vec<El<W>>>` → `f.maybe_effect(move |c, _| { ctx.set_children(c); })` — **Flex**.
- `#[layout(delegate = "field")]` → `Build::layout` returns `self.field.layout()` (the `Show` delegate, §5.4) — fleet.

Flags matching: the builder must report the widget's flags pre-build (they seed `ElState`). Options for the derive: (i) an attribute `#[flags(hoverable, clickable, focusable)]` on the builder, or (ii) a generated `Widget::flags` on the retained type plus a `_builder_flags()` const the builder forwards to. The prototype picks one and measures; (i) is less indirection.

**Prototype de-risking:** the plan hand-writes the two `Build` impls **first** (protocol + tests green with zero macro risk), then introduces `#[derive(Builder)]`, re-expresses Button/Flex through it, and **measures the LOC delta** hand-written→derived. If the field-attribute transform proves too large for the prototype window, the fallback is "derive the `View`/`SingleViewMarker`/`layout`/`flags` forwarders, hand-write `build`" — still a real ergonomics win, and the full derive becomes a 13.4 deliverable. Either way the *measured* boilerplate is a 13.3 gate input.

### 3.3 Robust ctx-param detection (7.7)

Replace the "push `'static` on every type param, assume a param literally named `W`" logic with: find the type param bounded by `WidgetCtx` (fall back to the first type param / a param literally `W`), name it explicitly in the emitted `View<Wctx>` / `Build<Wctx>` impls, and add `'static` only where the current hand-written impls do. This lets widgets whose ctx param is not literally `W` derive correctly and is a prerequisite for `#[derive(Builder)]` on generic builders (`Flex<W>`, later `Select<W, K>`).

## 4. De-generic `Flex`: `Dir` → `Axis` field (7.2, prototype slice)

`Flex<W, Dir: Direction>` carries `dir: PhantomData<Dir>` (`flex.rs:18-24`) — a build-only ZST *type param* the census flags. The `Direction` trait (`rsact-render/src/geometry/axis.rs:257-271`) maps `RowDir→Axis::X`, `ColDir→Axis::Y` at the type level, consumed once in `FlexLayout::base(Dir::AXIS, …)` (`flex.rs:52`).

De-generic to a runtime field:
```rust
pub struct FlexBuilder<W: WidgetCtx> { children: MaybeSignal<Vec<El<W>>>, layout: Layout, axis: Axis }
pub struct Flex<W: WidgetCtx> { layout: Layout }   // retained: just the layout node handle
impl<W> FlexBuilder<W> {
    pub fn row(children: impl ViewSequence<W>) -> Self { Self::new(children, Axis::X) }
    pub fn col(children: impl ViewSequence<W>) -> Self { Self::new(children, Axis::Y) }
}
```
`FlexExt::Row/Col` (`flex.rs:212-227`) and the `row!`/`col!` macros re-point to `FlexBuilder`. **Scope:** the prototype de-generics `Flex` only. `Select`/`Scrollable`/`Space`/`Bar`/`Slider` also carry `Dir` (grep in the analysis) and stay generic until 13.4 — they are independent generics, so leaving them is sound. This is the bounded slice of 7.2; the `V → numeric` half is entirely fleet.

Note this also lets `Flex` support runtime direction changes later (the `flex.rs:16` TODO) — a free side benefit, not a goal.

## 5. Per-widget migration recipe + the two conversions

### 5.1 The recipe (applies to every split widget)

1. **Split the struct.** Fields used only in `build` (children, construction-consumed config) → `FooBuilder`. Fields read in the hot loop (`layout`, `style`, `state`, closures) → retained `Foo`.
2. **Move the builder methods.** `new`, `LayoutWidget`/`SizedWidget`/`BlockModelWidget`/`FontSettingWidget`, and inherent setters (`gap`, aligns, `on_click`, `style`, …) → `impl FooBuilder`. `FooBuilder` gets `layout_mut()`.
3. **`#[derive(Builder)] #[builds(Foo<…>)]`** on the builder; annotate child fields.
4. **Move `fn build`'s body** into the generated/hand `Build::build` transform (consume children, construct `Foo`).
5. **Retained `Foo`** impls `Widget` (layout/render/on_event/update, flags) — no `build` override (default no-op), no builder traits, no `#[derive(View)]`.
6. **Keep every `Note:`/`TODO:` comment** (AGENTS.md) — move them with their code.

### 5.2 Button (single child) — the concrete target

- `ButtonBuilder<W> { layout, content: El<W> (#[child(single)]), style, on_click }` — carries `LayoutWidget`/`SizedWidget`/`BlockModelWidget`/`FontSettingWidget`, `new`, `on_click`, `style`.
- `Button<W> { layout, style, on_click }` — `Widget`: `layout`/`render`/`on_event` verbatim from today (`button.rs:86-114`); no `content` field.
- `Build::build`: `ctx.set_single_child(&mut content)` (was `Widget::build`, `button.rs:82-84`), construct `Button`.
- Win: drops the retained `content: El<W>` husk (`El::Stored{id, layout}` ≈ 2–3 words).

### 5.3 Flex (reactive children + de-generic) — the concrete target

- `FlexBuilder<W> { children: MaybeSignal<Vec<El<W>>> (#[children(reactive)]), layout, axis: Axis }` — carries `new/row/col`, `gap`, `wrap`, aligns, `center`, the Sized/BlockModel/Font/Layout builder traits, `FlexExt`.
- `Flex<W> { layout }` — `Widget`: `layout` returns `self.layout`; `render`/`on_event` are the existing no-ops (`flex.rs:189-199`).
- `Build::build`: `children.maybe_effect(move |c, _| ctx.set_children(c))` (was `Widget::build`, `flex.rs:183-187`), construct `Flex { layout }`.
- Wins: drops the retained `children: MaybeSignal<Vec<El<W>>>` (a whole `Vec` of husks + the `MaybeSignal` header) **and** the `dir: PhantomData` type param (§4). This is the larger of the two measurements.

### 5.4 Design-around cases (documented; deferred to fleet unless noted)

- **`Show` (layout delegate).** Owns no `layout` field; `Widget::layout` returns `self.el.layout()` (`show.rs:44-46`); `show` memo dead after `new`. When split, `ShowBuilder` uses `#[layout(delegate = "el")]` so `Build::layout` returns the child's layout; no `layout_mut`. **Not split in the prototype** — it stays unsplit and works via the derived identity `Build`.
- **Reactive structure stays on the build-time-effect path.** `Flex.children`, `Dynamic.current` (`dynamic.rs:55-64`), and the `Select` setter wire an *effect* in `build` that re-fires from the runtime subscription, not the field. The builder can drop the field post-wire; the effect (page-scope-owned) survives. Only reactive **props** are the newly-routed-through-build concern, and only in WS5.1 (§2.7). `Dynamic` needs no change for the prototype (unsplit; its build already uses `ctx`).
- **`Select`** (`select.rs`) — `Rc<MaybeReactive<Vec<SelectOption>>>` + Dir + `K` param; entangled with backlog **A18** (maintainer: "looks horrible"). Explicitly **fleet + coordinate with A18**, not prototype.

## 6. Measurement plan (drives 13.2) — with honest expectations

Measured on Button + Flex, before vs after the split. Tooling already exists (`metrics-probe/`): `charge()` (tracking-allocator heap bytes/allocs, `scenarios.rs:19`), `profile_counts()` (reactive node counts via `current_runtime_profile`, `scenarios.rs:52`), and `sizes.rs` (thumb `.text/.rodata` flash via the excluded `size-probe` crate).

| Axis | How | Honest expectation |
|---|---|---|
| **Static RAM (struct)** | `core::mem::size_of::<Button>()` vs `size_of::<ButtonBuilder>()`; same for Flex | Retained shrinks by the child husk (Button ≈ 2–3 words) / the `MaybeSignal<Vec<El>>` header + `PhantomData` (Flex). **Small** — analysis §1. |
| **Heap RAM (page)** | `charge()` on a page of N buttons / a Flex of N children, built then dropped-to-steady-state | ≈ neutral to slightly better — the child `Box<dyn Widget>` still lives in the arena; what's saved is the retained husk. |
| **Reactive nodes** | `profile_counts()` on the same pages | ≈ neutral in the prototype (props still bind via the node, §2.7); the node-count win is WS5.1's, which this enables. |
| **Flash** | `size-probe` `.text` thumbv7m, before/after | **Watch for a small increase** — the split adds a builder type + `Build` trait objects (more monomorphization). A regression here is a legitimate gate concern; quantify it. |
| **Ergonomics** | LOC of a split widget hand-written vs `#[derive(Builder)]`; call-site churn count | The derive must make the split *cheaper* than the status quo per widget, or the fleet isn't worth it. This is the decisive gate input. |

A near-neutral RAM/flash result is a **success**, not a failure: WS13's thesis (analysis §1, §5) is that it is an *enabler*, and the measurement's job is to confirm it does not *regress* the embedded floor (Blue Pill ≤ ~48 K flash / ≤ 20 K RAM) while delivering the protocol + acceptable ergonomics. Report all five axes to the gate even where the delta is ≈0.

## 7. Invariants & conventions respected

- **Arena↔layout structural parallelism** preserved (§2.6); `check_children_parallel` degrade-don't-panic unchanged.
- **"UI must never panic"** — the new `ElData::widget()` unbuilt-at-runtime path logs + returns `None`; no new `unwrap` on render/event/nav; the build-pass rebuild guard stays `error!` (`build.rs:86`).
- **Crate encapsulation** — `Build` lives in rsact-ui; rsact-reactive stays UI-vocabulary-free.
- **No feature-flag sprawl** — the split is architectural (pay-per-use by construction), gated by no feature.
- **Serial host tests** (`--test-threads=1`, `--features std`); `Note:`/`TODO:` comments moved with their code, never deleted (AGENTS.md, EVOLUTION protocol).
- **Reactive-setter regression** (`page/mod.rs:1817`, `:1845`) stays green — Edge/Label are unsplit, so `.width(sig)`/`.font_size(sig)` still bind through `layout_mut`.

## 8. Risks + 13.3 gate decision inputs

- **R1 — flash bloat from extra types.** The dominant risk. Mitigation: measure §6; if `.text` regresses, the gate weighs it against the WS5.1 enablement. (Builders are consumed/dropped, so the cost is code size, not steady RAM.)
- **R2 — derive complexity.** Field-attribute codegen for the transform is the largest new surface. Mitigation: hand-write first, derive second, measure the delta (§3.2); fall back to "forwarders-only derive" if needed.
- **R3 — arena accessor churn touches every pass.** `.widget` → `.widget()`/`widget_mut()` is wide but mechanical; covered by the existing 60 rsact-ui lib tests.
- **R4 — the win is deferred to WS5.1.** Not a bug, a framing risk: the gate must accept a near-neutral prototype on the strength of the *protocol* + ergonomics, per §6/§5's thesis.
